## 先读：一句话内核

本系统不是「固定流水线平台」（先分类意图再挂死节点），而是 **单 loop + 工具中心 + 流式事件 + 上下文压缩**：

```text
用户发消息
  → 控制面建一轮对话（Turn），执行面快速受理
  → 确定性预处理（斜杠命令可短路；多目标可给弱提示）
  → while 未超步数 / 预算:
        组装上下文 + 本步工具表
        流式调用模型（Function Calling）
        若有 tool_calls → 校验 / 审批 / 执行 → 结果写回消息
        若无 → 本轮结束
  → 事件推到 UI；旁路做索引重建、离线评价等重活
```

| 分工 | 谁负责 |
|------|--------|
| **意图** | 模型：选工具、填参数、写正文 |
| **事实** | Runtime：schema 校验、沙箱副作用、检索结果、引用核对、超时与取消 |
| **看见什么** | Context engineering：窗口预算下的裁剪与摘要 |
| **厚度** | Harness：入口、工具纪律、网关、护栏、可证明性——**加厚不许挡首 token** |

部署大致是：对外 API（鉴权、建 Turn、SSE）↔ Runtime（真跑 loop）↔ 工作区文件与数据库；前端只消费事件投影，不猜阶段。

## 术语说明（读问答前）

### Turn / Run / Step

| 词 | 是什么 | 为什么这样拆 |
|----|--------|--------------|
| **Turn** | 用户发一条消息触发的「一轮对话」 | 事件、取消、投影都按轮归档 |
| **Run** | 执行这一轮的一次运行实例（与 Turn 1:1） | 可中断、审批后继续；多副本时抢执行权 |
| **Step** | loop 里的一步：组装上下文 → 调模型 →（可选）跑工具 | 超时与过程可见性挂在 step 上 |

### 热路径 vs 旁路

| 词 | 是什么 | 为什么可行 |
|----|--------|------------|
| **热路径** | 用户点发送后、到首 token / 本轮结束，必须串行等到的路径 | 只放「小而必要」的事，否则交互卡死 |
| **旁路** | 异步任务、打字时后台预热、夜间抽样——**不挡**热路径 | 索引重建、大批量核查、内容打分放这里，用户仍感觉跟手 |

「检索热路径不建索引」= 搜索工具只加载已有索引并查询；全量切块与嵌入在资料变更后异步完成。索引未跟上时，搜索会 keyword 兜底，并提示索引可能滞后。

### 热文件路径（`hot_files`）

| | |
|--|--|
| **是什么** | 一份**短路径列表**（通常 ≤12 条工作区相对路径），表示「这轮会话最近在碰、或用户点名过的文件」 |
| **从哪来** | 用户消息里的 `@path`；入口预读成功后的路径；会话里记录的最近触及文件 |
| **怎么用** | 父 Agent 委派子 Agent 时，只把路径写进子任务说明；子 Agent 需要正文时再读文件 |
| **为什么可行** | 路径几十字 vs 全文几千 token；工作区是共享磁盘状态，指针够用。不必整包复制父对话历史 |

### `context_refs` / `paths`

委派时显式点名的文件指针，与 `hot_files` 同类。短备注可以附带，但大段资料应靠路径，而不是糊进一段自由文本（备注本身也有长度上限）。

### `@path` 预读

用户在输入里写 `@某个文件` 时，进模型前按**文件数 / 字符数 / 超时**预算读一小段贴进消息。超时则只留路径指针。用户已点名 → 确定性注入往往比「先搜再读」省一轮。

### 证据链（evidence）与 `unverified`

| | |
|--|--|
| **evidence** | **本轮**里通过检索或引用检查确认过的 `cite:…` id 集合 |
| **unverified** | 草稿里写了引用标记，但本轮从未捞到过该 id → 打标提醒，默认不硬阻断交付 |
| **为什么** | 「本轮是否捞到」是硬事实；幻觉可以被标脏，却不能靠机制从概率上消灭 |

### 素材卡 pin

写作场景里，角色 / 情节 / 风格等「卡」被选中后整段钉进本轮系统提示，优先级高于检索。卡是作者写死的约束；且默认不进 RAG 索引，避免被搜成噪声。

### `open_items` / 计划弱提示 / plan gate

见 **Q3**（含示例）。一句话：弱提示 nudge，规划是可选工具；未完成项跨轮带回；禁止「必须先计划再干活」。

### 工具白名单 / 阶段收缩

场景决定模型能看见哪些工具；交付后或步数已晚时，再从工具表拿掉检索 / 委派 / 记忆，减少空转与提示词体积。

### Tool-mediated（工具中介）

资料与偏好记忆不每轮预灌；只有模型调用对应工具后，结果才进对话。预灌会稳定占窗口、拖慢首 token。

### 离线 batch / 夜间抽样

对近期会话按低比例（默认 ≤5%）抽样，跑与 `/verify` 相同的确定性引用扫描，写报告。不改用户草稿，也不挂在每次交付末尾。用户要即时结果 → `/verify` 或「事实核查」按钮。

### 打字预热（warmup）

见 **Q12**：输入停顿约 300ms 后后台预热嵌入器与索引加载，不占发送后的 Turn 预算。

### 上下文压缩阶梯（fill ratio）

见 **Q9**：窗口占用比例升高时依次截断工具结果 → collapse → snip → autocompact。

### 混合召回 / RRF / 两级召回

词法 + 向量同时召回，用名次融合（RRF）；文档级与段落级可并行，超时则只保留段落通道。详见 **Q8 / Q13**。

### Stub

工具名、超时与并行脚手架在，真实业务后端未接，调用返回「未实现」。接第一张表时仍走同一工具，不新加固定编排节点。

### Golden / rubric / SLO

| 词 | 是什么 |
|----|--------|
| **Golden** | 固化「这一轮该怎样表现」的回归用例 |
| **Rubric** | 内容质量启发式打分，只离线抽样 |
| **SLO** | 体验红线：受理延迟、首 token、取消时延等 |

### Egress allowlist

模型 HTTP 请求只允许配置过的主机；未登记则拒绝。防止请求被指到未知第三方。

## Q0. 落地场景是什么？是否有用？

**结论：✅ 智能写作为主，通用沙箱 Agent 为辅；「检索 → 起草 → 引用检查」闭环已跑通。**

| 场景 | 在系统里意味着什么 |
|------|-------------------|
| 写作 | 资料检索、素材卡 pin、分节起草、导出、确定性引用核查（`/verify` 或「事实核查」） |
| 沙箱 Agent | 更宽的工具面；写文件 / 执行等走审批；按需记忆工具；多表结构化查询仍为 stub |

**有用判据**：跟手（延迟可控）、可控（审批与取消）、有据（检索与引用可核对）、可回归（golden / 指标）——不是「模型偶尔写得漂亮」。

**边界**：成文质量仍依赖线上模型；万档企业 ACL / 多租户未做；多表业务通道未接真实表。

## Q1. 是否经常幻觉或乱调工具？如何缓解？

**结论：🔸 幻觉无法消灭；用入口与证据链加压，而不是再问一轮模型。**

| 层 | 原理 | 现状 |
|----|------|------|
| 缩小动作空间 | 场景白名单；写作侧不开放危险 shell；后期可再砍检索 / 委派 / 记忆 | ✅ |
| 入口失败 | 参数不合 JSON Schema → 结构化错误回灌，不进真正执行 | ✅ |
| 证据链 | 本轮 evidence；草稿 cite 不在集合 → `unverified`（不硬阻断） | ✅ |
| 人机核对 | `/verify` 即时扫；覆盖面靠离线 batch，不挂交付尾 | ✅ |
| 可观测 | 工具误用 / schema 失败计数 | ✅ |

机制能标、能查；不能从概率上抹掉胡说。零命中时不编 cite，仍部分依赖模型遵守提示。

## Q2. 怎么评价 Agent 系统好坏？

**结论：✅ 分层评价；内容质量分永不挂在用户 Turn 尾。**

| 层 | 测什么 | 何时跑 |
|----|--------|--------|
| 行为契约 | 该调的工具、该守的格式（golden） | CI / 本地 |
| 体验 SLO | 受理延迟、首 token、取消时延 | 在线可度量 |
| 工具健康 | misuse、schema 失败率 | 在线 |
| 检索 | 混合召回是否有 hit、索引是否滞后 | 离线 / 专项评测 |
| 内容质量 | 抽样启发式打分 | **离线**，不拖交付 |

原则：在线证明「跟手与可控」；「写得好不好」离线抽样，避免为打分付交互税。

## Q3. 任务规划如何保证拆解合理？

**结论：🔸 规划是可选能力，不是强制阶段。系统给弱提示与跨轮备忘，不替模型「先开一遍计划会」。**

### 为什么不加强制 plan gate？

若规定「任何任务必须先调用规划工具，否则不许干别的」：

- 一句「把大纲第三点改短一点」也要多付一整轮模型调用；  
- 用户感觉发呆，却看不到正文进展；  
- 违背「热路径只放必要工作」。

所以：**简单任务可以零规划；复杂任务可以规划，但不强制。**

### 系统具体做什么？

**1）入口弱提示（nudge，非命令）**

用户消息看起来「像多个目标」时，往运行时上下文塞**一行建议**（大意：多目标时可先更新计划，可选）。判断是确定性的启发式，例如：

- 出现较多编号条目（1. 2. 3. 或 `-` 列表多项）；或  
- 出现多个承接词（「然后 / 并且 / also / finally」一类）且篇幅不太短。

模型可以无视；简单请求通常根本不会触发。

**示例 A — 会触发弱提示**

> 请做三件事：  
> 1. 根据资料写张白鹿出场  
> 2. 补一段环境描写并带引用  
> 3. 导出到 exports  

入口识别到多条编号目标 → 上下文里多一行「可考虑先更新计划」。模型可能先列 to-do，再检索 / 起草；也可能直接开干——都合法。

**示例 B — 通常不触发**

> 把第二节标题改成更短的。  

无多目标形态 → 不塞计划提示；模型改标题即可，不必先 `update_plan`。

**2）规划工具本身（可选）**

模型可调用「更新计划」类工具，写出有序待办（待做 / 进行中 / 完成）。这只是普通工具结果，写进对话，方便自己和下一步看见，**不是**单独阶段节点。

**3）跨轮 `open_items`**

本轮正常结束后，系统从对话里抽出**尚未完成**的计划项，缩成短列表写进会话摘要（有上限）。下一轮若仍带着这份会话上下文，模型能看到「上次还没做完什么」，而不必把整段计划对话原文再灌一遍。

**示例 C — 跨轮**

- 第一轮计划了：写出场 → 补引用 → 导出；只完成了出场。  
- 会话里留下 open_items：如「补引用」「导出」。  
- 用户第二轮说「继续」→ 模型有机会接着未完成项，而不是失忆。

### 合理拆解靠什么「保证」？

严格说：**不能数学保证**。能做到的是：

| 手段 | 作用 |
|------|------|
| 弱提示 | 复杂任务更容易先拆再干 |
| 可选规划工具 | 拆解过程对用户 / 事件时间线可见 |
| open_items | 跨轮不忘尾巴 |
| 步数与预算 | 防无限拆、无限搜 |

不保证的：模型仍可能漏步、乱序；这靠产品提示词、用户打断、以及后续核查，而不是强制门。

## Q4. 子 Agent 之间上下文怎么传递？

**结论：✅ task + 文件指针进，summary 出；禁止整包消息历史共享。**

### 父 → 子带什么？

1. **task**：要做什么（短文本）。  
2. **短备注**（可选）：几句说明，有长度上限，不适合贴整篇资料。  
3. **`context_refs` / 路径**：本次点名盯住的文件。  
4. **`hot_files`**：父会话汇总的「最近相关路径」≤12 条。

子 Agent：工具更少、深度有限（最多再套两层）、步数更短；正文按需读文件。父对话的完整 messages **不**整包拷贝过去——既省窗口，也缩小权限与干扰面。

### `/verify` 或「事实核查」出来的结果代表什么？

这是**确定性扫描**，不调模型，也**不改**草稿正文。系统会：

1. 收集工作区里最近的草稿 / 分节 / 导出 Markdown；  
2. 抽出文中的 `[cite:…]`（含中文 id）以及形如 `sources/…`、`sections/…` 的路径引用；  
3. 逐条问：「对应资料文件是否真的存在 / 能对上？」；  
4. 写一份报告落到工作区（如 `.agent/verify-reports/`），并向用户返回摘要。

| 字段（含义） | 代表什么 |
|--------------|----------|
| **checked** | 一共核对了多少处引用或路径 |
| **invalid** | 其中有多少对不上（资料不存在或路径无效） |
| **findings** | 明细：哪份草稿里的哪个 cite / 路径，是否 valid |
| **report_path** | 完整报告文件路径，可打开复查 |
| **mutated_draft: false** | 明确：本次**没有改写**你的稿子 |

**怎么读结论**：

- `checked ≥ 1` 且 `invalid = 0`：文中出现的引用标记，在资料库侧都能对上——这是「引用锚点存在」，**不是**「论述事实已由人审过」。  
- `invalid > 0`：有 cite 或路径悬空（可能编了、写错了、或文件已删/未放入 `sources`）。  
- `checked = 0`：草稿里几乎没有可扫的引用形态——谈不上通过或失败，只是「无可核项」。

与写作过程中的 `unverified` 标记互补：后者看「本轮 Turn 是否检索过」；`/verify` 看「磁盘上引用是否指得到文件」。两者都**不**判断「这句话在历史学上是否正确」。

### 离线 batch

同一套扫描逻辑，对近期会话低比例抽样跑，用于覆盖面与回归，不挡交互。详见术语「离线 batch」。

## Q5. 是否使用 ReAct？优缺点？

**结论：✅ 主循环 = `while` + Function Calling，即工程上的 ReAct。**

| | |
|--|--|
| **优点** | 灵活选工具、可中途取消、步骤可见 |
| **缺点** | 步数膨胀、重复搜索、假参数 |
| **抑制** | 预算、schema 门、只读结果缓存、超时与卡住巡检、后期收缩工具表 |

外层编排框架若存在，也只是薄封装；**循环语义在引擎内**，不靠多节点业务图表达写作流程。

## Q6. 模型捏造工具参数怎么处理？

**结论：✅ 假参数在真正执行前被确定性拦住；失败信息结构化回灌，让模型改，而不是让错误副作用发生。**

### 问题长什么样？

模型「会说话不会填表」时常见：

- 必填字段缺失（检索没给 `query`）；  
- 类型不对（`limit` 给了字符串）；  
- 编造枚举外的模式；  
- 路径 / id 乱填。

若直接执行：轻则空跑，重则写错文件、误导出。

### 处理流水线（直觉版）

```text
模型发出 tool_call（工具名 + 参数 JSON）
  → 若该工具需要人批准且尚未批准：暂停，等人点允许/拒绝（不执行）
  → 否则用该工具的 JSON Schema 做校验
        通过 → 在超时限制内执行真正逻辑
        失败 → 返回 invalid_arguments（缺什么、哪错了、期望什么）
              → 当作工具结果写回对话 → 模型看见后可改参再调
```

### 失败时用户 / 时间线会感觉到什么？

不会出现「文件已被错误参数写坏」。通常会看到一次工具失败摘要，大意是：某工具拒绝了非法参数，并列出缺失字段或校验细节。模型下一跳有机会補正；若模型坚持胡说，可能反复失败，直到步数用尽——这比静默写坏更安全。

**示例**

1. 模型调用检索，但漏了查询词 → 校验失败：缺少必填 → 不查库。  
2. 模型补上查询词再调 → 通过 → 返回命中列表。  
3. 模型调用写文件，参数齐全但内容像 API Key → 那是 **Q11 secret 扫描** 的事（校验通过后、写入前），与 schema 门不同层。

### 和「再问模型：你参数对吗？」的区别

| 做法 | 代价 | 本系统 |
|------|------|--------|
| 用 LLM 复核参数 | 多一轮延迟，仍可能错 | ❌ 不用 |
| Schema 确定性校验 | 毫秒级，可重复 | ✅ 默认开启 |
| 人审批高风险写操作 | 交互一次确认 | ✅ 写 / 执行类 |

口诀：**参数形状由 schema 守门；是否允许副作用由审批与扫描守门；语义对不对仍靠模型与证据链。**

## Q8. 一万个长文档做 RAG，怎么解决？

**结论：🔸 小库～千档的「查得快、不卡交互」路径已齐；真正企业万档（权限、分片、运维）未完。**

### 现在已经具备的扩库前置

| 环节 | 做法 | 用意 |
|------|------|------|
| 热路径 | 搜索只加载索引 + 查询，禁止同步全量重建 | 用户不等建索引 |
| 旁路索引 | 资料变更后异步切块 / 嵌入；空索引 → 文件 keyword 兜底并提示可能滞后 | 正确性可降级，延迟优先 |
| 存储 | 默认可文件库；规模上升可切到 Postgres + pgvector（近似最近邻） | 同工具面换后端 |
| 召回 | 词法 + 向量 + RRF；文档级与 chunk 级可并行，超时降级 | 专名与语义兼顾，且控延迟 |
| 重排 | 轻量词法重排默认可开；交叉编码器默认关 | 重模型不挡交互 |

### 当前局限（诚实账）

| 局限 | 意味着什么 |
|------|------------|
| **无多租户 ACL** | 不能按「用户 A 看不见用户 B 的文档」做检索过滤；偏单人 / 单工作区自用 |
| **词法不是 Elastic 级** | 自研 / 轻量 BM25 + 文件兜底，不是分布式倒排集群；万档高并发、复杂查询语法未覆盖 |
| **索引运维偏简** | 有异步同步与滞后提示，但缺企业级增量水位、多副本一致性、热更新 SLA 面板 |
| **切块偏规则** | Markdown 标题 + 定长滑窗；无大规模「语义再切分」离线流水线 |
| **重排默认偏轻** | 交叉编码器默认关，相关性极端场景可能不如重模型精排 |
| **评价与灰度** | 有检索评测入口，但缺万档规模的持续回归集与灰度发布体系 |

写作向「几百～上千篇资料、单工作区」可以自用；「一万长文 + 部门权限 + 审计」还不能当成已交付能力。

### 若朝着企业万档继续做，建议顺序

不必推翻 loop，按层加厚：

1. **权限进检索**：文档元数据带 owner / 租户 / 标签；查询强制 ACL 过滤后再排序（召回与过滤同事务语义）。  
2. **索引分层**：热库 ANN（pgvector 或专用向量服务）+ 词法倒排（ES / OpenSearch）；旁路增量索引，热路径只读副本。  
3. **两级召回产品化**：先 doc 再 chunk 的超时与降级做成稳定 SLA；监控 `index_lag`、空结果率、超时降级率。  
4. **切块升级（旁路）**：保留标题切；对超长节做语义切或结构感知切；变更只重嵌受影响文件。  
5. **精排可选**：对 top-N 开交叉编码器或业务精排，严格超时，失败回落词法分。  
6. **评测闸**：固定万档抽样查询集；上线前门禁 hit@k、延迟分位、ACL 负例（不该召回的绝不出现）。

整条原则不变：**扩能力进旁路与可降级路径，禁止让「建索引 / 重精排」回到用户第一次搜索的热路径。**

## Q9. Context / Prompt / Harness Engineering？

**结论：✅ 三层分工清楚，也是理解本项目的主框架。**

| 层 | 管什么 | 本项目怎么做 |
|----|--------|--------------|
| **Context engineering** | 模型**看见什么**、窗口是否爆 | 组装系统提示、项目说明、运行时块、会话消息；按占用比例阶梯压缩；工具结果先截断 |
| **Prompt** | **话术与角色** | 场景静态系统提示；弱计划提示、素材 pin、运行时块属动态 |
| **Harness** | 包住模型的**运行时厚度** | 见下与 Q14 |

### fill ratio → collapse → snip 是什么意思？

把上下文窗口想成一杯水：

- **容量** ≈ 模型上下文长度减去「留给模型继续往外写」的预留。  
- **fill ratio（填充率）** ≈ 当前准备发给模型的内容估算 token ÷ 容量。  
- 水涨上来，不能让杯子溢出去（溢出轻则截断失败，重则请求直接报错），所以按水位抬级处理：

| 水位（约） | 策略名 | 人话 |
|------------|--------|------|
| 平常就做 | 工具结果截断 | 单次搜索 / 读文件回传太长，先砍到预算内 |
| ~80% | **collapse（折叠）** | 保留最早的用户意图 + 最近热区对话；中间大段换成「这里有过一段历史，已折叠」类指针，不再逐字携带 |
| ~90% | **snip（剪掉）** | 再不够，就丢掉最旧的一批完整消息（整段删，而不是糊成指针） |
| ~95% | **autocompact（自动摘要）** | 把大段历史收成一段摘要再塞回去；可用较小模型做摘要，失败则用规则摘要 |

所以「fill ratio 触发 collapse → snip」= **占用越高，丢失历史的方式越狠**：先折叠中间，再硬删旧消息，最后摘要保命。目的都是：当前任务还能继续，而不是为了「完整聊天记录」把请求打爆。

### 「Harness 厚度」怎么理解？

把裸模型想成发动机；**Harness（挽具）**是车上所有让车能上路的系统。  
**厚度**不是「代码行数多」，而是：**在模型周围叠了多少层可靠机制**，让「能聊天」变成「能交付的 Agent」。

| 更薄（几乎裸调） | 更厚（本仓库方向） |
|------------------|-------------------|
| 把用户话直接 POST 给模型 | 发送前：斜杠短路、@文件预读、场景工具白名单 |
| 模型说啥是啥 | 参数 schema、写操作审批、secret 扫描 |
| 窗口满了就报错 | fill ratio 阶梯压缩 |
| 卡住只能刷新页面 | 取消、步超时、卡住巡检 |
| 出问题靠感觉 | 事件、golden、延迟指标、离线抽样核查 |

厚度的约束是 **「厚不许挡快」**：每一层都要能取消、能超时、能降级；不能为了「更安全」让首 token 永远出不来。详见 **Q14**。

## Q10. Tool 与 Memory？长短期怎么分？

**结论：🔸→趋✅ 工具协议完整；显式记忆仓已接，是否召回仍由模型选用工具决定。**

| 寿命 | 机制 |
|------|------|
| 极短 | 当前轮消息 + 工具结果截断 |
| 短 | 会话记录 / 摘要 / open_items |

## Q11. 隐私消息如何防泄漏？

**结论：✅～🔸 出口与写出有内容级防护；多租户与磁盘字段级加密未做。**

### 分层在防什么？

| 面 | 做法 | 防的是什么 |
|----|------|------------|
| 结构边界 | 工具只能碰工作区沙箱路径；内网调用带服务 token；密钥加密存放 | 乱读系统盘、假冒内部服务 |
| 出站白名单 | 模型请求的主机必须在允许列表，否则直接拒绝 | 把对话拐到未知第三方 API |
| **脱敏（redact）** | 见下 | 手机号等敏感串随请求/日志外泄 |
| **secret 扫描** | 见下 | 密钥被写进工作区文件再扩散 |
| 未做 | 多租户隔离、静态字段级加密 | 企业租户场景 |

### 脱敏（redact）脱敏什么？

在消息**即将发给模型**之前，以及对结构化日志里若干文本字段，用**正则**替换高置信敏感串（不用 LLM 做脱敏，以免又慢又不可复现）：

| 模式类 | 被替换成 |
|--------|----------|
| 中国大陆手机号形态 | `[REDACTED_PHONE]` |
| 18 位身份证形态 | `[REDACTED_ID]` |
| OpenAI 风格 `sk-…` | `[REDACTED_API_KEY]` |
| AWS `AKIA…` | `[REDACTED_AWS_KEY]` |
| PEM / OpenSSH 私钥块 | `[REDACTED_PRIVATE_KEY]` |

注意：这是「发给模型 / 打日志前的掩护」，不是对用户稿件的持久改写；也不是万能 PII 引擎（地址、姓名等未全覆盖）。

### secret 扫描扫描什么？

在 **写文件 / 导出文档** 真正落盘前，扫描**即将写入的文本**，盯的是「高置信密钥形态」（集合刻意很小，以免散文误伤占满时预算）：

- AWS Access Key（`AKIA…`）  
- OpenAI 风格 `sk-…`  
- `BEGIN … PRIVATE KEY` 私钥头  
- GitHub PAT / OAuth token（`ghp_` / `gho_` 等）

命中且未超时 → **拦截本次写入**。扫描有毫秒级同步预算（默认约 50ms）：预算内扫完为准；若超时，先放行写入并异步再扫一遍（偏可用，仍打日志）——避免大文件扫描堵死交互。

与脱敏的分工：**脱敏**管「出站给模型 / 日志」；**secret 扫描**管「写进磁盘会不会留下钥匙」。

## Q12. 保证效果前提下如何加速？

**结论：🔸 加速服从「不挡受理 / 可取消 / 可降级」。**

| 杠杆 | 作用 |
|------|------|
| 快路径受理 | 先受理再异步跑 loop，token 流式推 UI |
| 只读并行 + 缓存 | 列表 / 读文件 / 检索等同参可合并 |
| **打字预热** | 见下 |
| 摘要小模型 | 高压摘要可走便宜模型，失败回退规则摘要 |
| 晚步缩工具 | 交付后或步数已晚，拿掉检索 / 委派 / 记忆 |
| 检索不建索引 | 热路径只查 |

### 打字预热：具体怎么做？预热什么？

目标：把「第一次检索要冷启动加载模型 / 读索引」的成本，挪到用户**还在打字**的时候。

**前端逻辑（示意）**

1. 用户在输入框改文字 → 更新本地状态；  
2. 若有未触发的定时器，先清掉；  
3. 若内容非空，启动约 **300ms** 的 debounce；  
4. 300ms 内又敲键 → 重新计时；停顿满 300ms → 发一次「预热」请求（把当前输入前缀带上，有长度上限）；  
5. 接口立刻 **202 接受**，不阻塞输入框。

**后端预热实际干两件事（后台任务，失败只打日志）**

1. **跑一遍嵌入**：用当前前缀（或占位词）调用嵌入器 `embed`——迫使句向量模型完成加载 / 进缓存；  
2. **加载资料索引到进程内**：把检索用的 store `load` 一遍——避免发送后第一次 `search` 再付磁盘反序列化成本。

预热**不做**：完整语义搜索排序、改索引、也不创建 Turn。  
因此它是「发消息前的旁路」；发送后的热路径仍按正常 loop 走，只是更容易「已经热乎」。

进程启动时往往也会先 warmup 一次嵌入器，与打字预热是同一类目标、不同时机。

## Q13. 为什么混合召回？Chunk 怎么切？

**结论：✅ Tool-mediated RAG + 混合召回 +「标题优先、过长再滑窗」的切块。**

### 为什么混合？

- 专名、角色名、作品名：词法更稳；  
- 同义改写、场景描述：向量更稳；  
- 向量挂了或索引空：退回工作区文件 keyword，保证「还能搜」。

两路结果用 **RRF（按名次融合）** 合并，避免两套分数强行比大小。需要时还可文档级与 chunk 级并行，超时则只用 chunk 通道。

### 我们具体怎么切？

对 `sources` 下要索引的文件（跳过点文件、调试贴、以及 `cards` 目录——卡走 pin，不走 RAG）：

1. **按 Markdown 一级～三级标题**切成「节」；节标题会拼进该节文本，方便命中。  
2. 若某一节仍过长：按 **400 字符** 一段、**重叠 80 字符** 滑窗切开（按字符串长度，不是按词）。  
3. 每块带上路径、节标题、行号范围、`cite:文件名主干` 作为引用 id，再嵌入存库。

检索命中后回到模型的是短摘录 + 引用 id；需要通读再 `read_file`。这就是 Tool-mediated：库很大也不预灌全文。

### 400 / 80 算大还是算小？

| 视角 | 判断 |
|------|------|
| **对中文网文 / 设定集** | 约一小段到一小节：多数标题节不触发滑窗；触发时 400 字仍偏「局部证据」，适合挂 cite，避免一块塞进半章 |
| **对英文** | 约 60～100 word 量级，偏细，有利于定位，但块数变多、索引更大 |
| **偏小的风险** | 证据碎片化，模型可能要多搜几次或再 read；重叠 80 用来减轻「句子被切断」 |
| **偏大的风险** | 单 hit 噪音多、引用粒度变粗、嵌入与重排更贵 |

当前取值是 **写作资料、要引用落点** 导向的偏细默认，不是通用「越大越好」。若资料多为无标题的超长纯文本，会更常走 400 滑窗，块数上升——这是已知局限；要改善应在**旁路**加结构 / 语义再切，而不是把块疯涨到热路径里重切。

经验上：标题结构良好的库，400 很少成为瓶颈；体感「找不到」时更优先查索引是否滞后、混合是否开启、ACL（若未来有）是否过滤过头，而不是先盲改 chunk 到几千字。

## Q14. Harness 是什么？

**结论：✅ Harness = 决定「好不好用」的运行时厚度，不是换模型，也不是加 workflow 节点。**

「厚度」的直觉见 **Q9**。六面可以记为：

```text
入口（Intake）     确定性编译输入；斜杠短路；@path 预算预读
上下文（Context）  组装与压缩；报告填充率与组装耗时
工具（Tools）      白名单、审批、超时；只读并行；schema 门
模型（Model）      统一网关：生成参数、重试分类、首字节超时、出站脱敏
护栏（Guard）      取消贯串；步 / 工具超时；卡住巡检；出站白名单
证明（Proof）      事件字段、golden、延迟门禁、离线打分 / 抽样核查
```

**铁律：厚不许牺牲快。**

## Q15. 多数据表下如何稳定召回？

**结论：🔸 接口与超时脚手架在；真实业务表与 ACL 未接（stub）。**

设计姿态：结构化查询仍表现为**一个工具**（内部可规则路由多通道、通道超时），**不**为此新增长固定编排图。  
现状：调用返回未实现。产品触发时接第一张带权限的表，复用同一工具契约：查询进、行级结果出、失败可降级。

稳定召回要点（接表时要对齐）：每通道超时预算、租户过滤与查询同层、空结果与超时要可区分、禁止在工具里同步全表扫描。

## Q16. 动态 Prompt 与静态 Prompt？

**结论：✅ 能静不动；动态块可截断、有预算。**

| 类型 | 例子 |
|------|------|
| 静态 | 场景系统提示、工具名称与描述与参数 schema |
| 动态 | 运行时上下文、计划弱提示、素材 pin、压缩后的会话尾 |

## Q18. 为什么要区分静态 / 动态长期记忆？

**结论：🔸「钉子」钉规则，「滚窗」扛会话。**

| | 静态倾向 | 动态倾向 |
|--|----------|----------|
| 内容 | 角色、风格、硬规则 | 会话摘要、open_items、remember 笔记 |
| 注入 | 高优先级前缀 / pin | 工具召回或滚动压缩 |

静态保跨轮一致性；动态防窗口被历史撑爆。

## Q19. 端到端链路怎么运转？

**结论：✅**

```text
Web 发消息
  → 控制面创建 Turn / Run
  → 执行面快速受理
  → 入口：/help · /compact · /verify 可短路（无模型）
  → while step:
        组装上下文 +（可阶段收缩的）工具表
        模型流式输出 + Function Calling
        执行器：审批 → schema → 执行（检索只查不建；写前 secret 扫）
  → 结束 → 投影 / 会话摘要 / 可选 open_items 回填
  → 事件流 → UI
```

旁路（不挡首 token）：索引异步任务、打字预热、离线抽样核查、离线质量打分。

## Q20. Function Calling 怎么运作？

**结论：✅**

1. 场景 → 工具白名单 →（后期可再缩）工具表发给模型  
2. 模型返回 tool_calls  
3. JSON Schema 校验  
4. 写 / 执行类审批  
5. 执行：检索热路径不建索引；写出前 secret 扫  
6. 结果回灌；引用进入 evidence 或标 unverified  

口诀：**模型出意图，Runtime 出事实。**
