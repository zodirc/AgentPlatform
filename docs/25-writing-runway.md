# 25 — Plan 模式（平台层：先规划、同意、可见执行）

> **状态：P0 落地中** · 2026-07-20 · 平台层 Plan（writing / agent / interview）  
> 关联：[`09`](09-product-modes.md) · [`10`](10-product-experience.md) · [`06`](06-tools-and-context.md) · [`08`](08-event-projection-pipeline.md) · [`14`](14-writing-quality.md) · [`23`](23-writing-work-model.md) · [`24`](24-writing-token-economy.md) · [ADR-015](adr/015-interrupt-cancel-resume.md) · [ADR-014](adr/014-turn-intake-over-intent-pipeline.md) · [R1–R5](13-rate-redlines.md)  
> **范围：全场景 workbench（writing / agent / interview），不是写作专属。**  
> 本文回答：多步任务如何像 Cursor Plan 一样——规划、同意、执行中看见每一步是否完成——且不毁默认短交互。

**已落地（P0）：** `turn.plan` 投影以最新为准；共享 `PlanPanel`；聊天区 Plan 按钮 + 多目标建议条；「按此执行」**仅**在 Plan 模式回合产出「全 pending」清单后出现；任一步变为进行中/完成后隐藏，避免重复执行。  
**未做（P1+）：** 硬禁止规划期写正文、同 Run 同意 interrupt、自动续 Turn。

---

## 1. 结论先讲

1. **Plan 是平台交互能力**，写作只是受益场景之一；agent 改多文件、访谈待办同构。  
2. 用户要的不只是「有个计划」，而是 Web 上 **步骤清单 + 当前步 + 哪步已完成**（过程可见，对齐 `10` 北极星）。  
3. **入口形态对齐 Cursor / Copilot 级产品：**  
   - **主路径**：显式控件（Plan 按钮 / 模式切换）——用户主动进轨 B；  
   - **辅路径**：普通 Agent 模式跑着跑着，**仅在少数时刻建议**「这件事更适合先 Plan」——用户点头才切，**禁止静默变轨**。  
4. 默认永远是普通 Agent；Plan 是稀缺模式，不是默认人格。  
5. 我们已有半成品（`update_plan` → `turn.plan` → artifact），缺的是 **入口 UX、同意门、全场景 Plan 面板、执行中状态刷新**。  
6. **静默连写 / 常开长跑 / 每 Turn 强制 Plan / 系统擅自切 Plan** 仍否决。

---

## 1.5 入口：按钮为主，建议为辅（关键）

成熟 IDE / Copilot 类产品里，Plan **不是**「模型觉得复杂就自己改协议」，而是：

```text
┌─────────────────────────────────────────┐
│  普通 Agent（默认，几乎所有时间）          │
│  跟手改、问答、单点工具                    │
└───────────────┬─────────────────────────┘
                │
     ┌──────────┴──────────┐
     │                     │
 ① 用户点「Plan」        ② 普通模式里偶发提示
    （显式按钮/切换）        「建议先做个计划 → [切换]」
     │                     │ 仅当多文件/多目标/高风险
     │                     │ 用户点切换才进入
     └──────────┬──────────┘
                ▼
         Plan 模式（轨 B）
         规划 → 同意 → 可见执行
```

| 入口 | 谁发起 | 是否自动改行为 |
|------|--------|----------------|
| **Plan 按钮** | 用户 | 是——进入后才按 Plan 协议 |
| **「建议切换到 Plan」** | Agent 提议，**用户确认** | 否——不点则继续普通模式 |
| ~~模型自行进入 Plan 并开长跑~~ | — | **禁止** |
| ~~每个请求都先 Plan~~ | — | **禁止** |

### 何时才「建议切换」（宜极窄）

普通模式 Agent **可以觉得**「转 Plan 更佳」，但只应在少数时刻冒泡，例如：

- 单条请求里 **多个独立目标**（已有 intake 弱提示基因，升级为「可点的建议」而非闷头 `update_plan` 开写）  
- 将触及 **多文件 / 长工具链 / 难撤销** 的改动  
- 用户原文就在要「先规划 / 给个方案再动手」

**不要**建议的时刻：改一句文案、问个短问题、单章小补丁、「继续」精修——保持普通模式跟手。

### 建议的形态（少信号）

- UI：**一条可关闭的提示条** +「切换到 Plan」按钮；不是新 Scenario，不是改 system 人格分裂。  
- 文案：建议切换，**不**自动 `update_plan` 后开写。  
- 触发：规则 / 显式用户话术优先；**禁止**为「像不像该 Plan」再同步调一个分类模型（R2 / ADR-014）。  
- 频率：同一会话冷却，避免刷屏催切。

### 和「同意门」的关系

- **进 Plan**（按钮或采纳建议）≠ **同意执行计划**。  
- 进 Plan 之后仍是：先出步骤 → 用户再点「按此执行」。  
- 两道门：模式门（要不要 Plan）+ 契约门（要不要按这张清单干）。

## 2. Agent 交互逻辑（应有的心智）

### 2.1 两条轨道，不要搅成一条

```text
轨道 A — 默认对话（日常）
  用户一句 → 短 Turn → 直接做事或回答
  无计划板，或计划板空
  体感：跟手、可停、可挑剔

轨道 B — Plan 会话（多步）
  进入 Plan（显式）
    → 产出步骤指南（可见、可改）
    → 用户同意
    → 按项执行（清单状态滚动）
    → 可随时 Stop；已完成项保留
  结束后回到轨道 A 做单点精修
```

**破坏体验的混用：**

- 轨道 A 被 **静默** 升级成 B（无按钮、无确认的「自动 Plan」）  
- 轨道 B 没有同意门（假 Plan，列完直接干）  
- 轨道 B 没有可见进度（用户只看见「模型思考中」）  
- 所有请求强制 B，或普通模式动不动弹「请切 Plan」刷屏  
- 把「建议切换」做成后台已按 Plan 长跑（建议必须是 **可点、可忽略**）

### 2.2 三层「可见」，别混

| 层 | 是什么 | 回答用户哪句话 |
|----|--------|----------------|
| **活动态** | 思考中 / 跑某工具 / 待审批 | 「此刻机器在干什么？」 |
| **计划态** | 步骤 1…n 的 pending / in_progress / done | 「整件事走到哪了？」 |
| **产物态** | diff、草稿章、检索命中 | 「改出了什么？」 |

成熟产品三层都在；我们今天 **活动态有、产物态有、计划态半残**（agent/interview 有简陋列表，writing 侧栏甚至不展示；且多次 `turn.plan` 投影是 append，UI `.find` 可能不是最新）。

### 2.3 计划项生命周期（交互契约）

```text
pending → in_progress → completed
                ↘ cancelled / skipped（用户 Stop 或改计划）
```

约定（少信号、可测）：

1. **进入执行前**：清单已存在且用户已同意（或用户消息明确「按此执行」）。  
2. **开始某步**：该步 → `in_progress`（应再调 `update_plan` 或等价投影更新）。  
3. **完成某步**：→ `completed`；Web 立刻可见。  
4. **Stop**：当前 `in_progress` → `cancelled` 或留 in_progress 但 Turn `cancelled`；**已 completed 不回滚**。  
5. **不**为每步强制再开评审模型。

### 2.4 同意门（与工具审批不同）

| | 工具审批 | Plan 同意 |
|--|----------|-----------|
| 对象 | 一次高风险工具 | 整份步骤契约 |
| 用户问题 | 「允许跑这条命令/写这个文件吗？」 | 「按这张清单做吗？」 |
| 之后 | 继续同一 Run 或否决 | 开始/拒绝多步执行 |

推荐落地序：

- **先两 Turn**：Turn1 只规划并展示；Turn2「按这个执行」。  
- 再考虑同 Run `waiting_plan_approval`（状态面更大）。

## 3. 成熟 agent 在这方面怎么做

对照 Cursor Plan、带 TODO 的编码 agent、类 Devin 的清单执行（抽交互，不抄 UI 皮）：

| 做法 | 含义 | 对我们的启示 |
|------|------|----------------|
| **Plan 是模式/按钮** | 用户看见并主动切换 | workbench 上明确的 Plan 入口，不是藏在 prompt 里 |
| **普通模式可建议切换** | Agent 觉得多步更佳时提议，人决定 | 窄条件提示条；不自动变轨 |
| 默认仍是普通 Agent | 绝大多数时间跟手 | Plan 稀缺；冷却、不刷屏 |
| 计划是一等公民面板 | 侧栏清单常驻 | **共享 PlanPanel**，三场景都挂 |
| 步骤状态滚动更新 | 当前步高亮、完成打勾 | `update_plan` 执行中刷新；投影最新一份 |
| 先展示再跑 | 同意前少改世界 | 规划阶段不写正文 |
| 执行可打断 | Stop 清晰 | Cancel + 清单反映停点 |
| 不替代 diff/产物 | 计划 ≠ 变更审阅 | 计划旁仍要 patch/草稿 |

它们通常 **不** 做：无确认自动进 Plan；无面板后台长跑；每个小改动先逼你 Plan；为「该不该 Plan」再挂同步分类模型。

## 4. 现状缺口（代码事实）

| 已有 | 缺口 |
|------|------|
| `update_plan(items[{id,title,status}])` | 模型不保证执行中刷新 status |
| `turn.plan` → artifact `type=plan` | projector **append** 多份；UI 常 `.find` 第一份 → **进度易脏** |
| agent / interview 有简陋计划列表 | **writing 侧栏无计划**；无「当前步」强调 |
| 多目标 intake 弱提示可 `update_plan` | **无同意门**；无「仅规划/执行中」模式条 |
| `open_items` 回填会话摘要 | 不等于 Turn 内实时清单 |

因此：不是「从零发明 Plan」，而是 **把半成品做成平台级过程可见**。

---

## 5. 写作只是场景之一

| 场景 | 计划项例子 | 执行时用户要看见 |
|------|------------|------------------|
| writing | 确认节拍 → 起草第 N 章 → 起草第 N+1 章 → 停让你改 | 哪一章在写、哪章 done |
| agent | 读 A → 改 B → 跑测 → 修失败 | 哪步在跑命令/改文件 |
| interview | 问主题1 → 主题2 → 收束 | 访谈待办勾选（已有雏形） |

静默「连续写几章」仍否决；若多章出现，只应作为 **已同意计划里的若干步**，且每步在面板上可勾选进度。

## 6. Web 应长什么样（产品要求）

共享组件（建议挂 workbench，而不是写三个副本）：

1. **计划标题 + 摘要**  
2. **步骤列表**：状态图标（待办 / 进行中 / 完成 / 取消）  
3. **当前步**视觉强调（与「模型思考中 · 第 N 轮」互补，不互相替代）  
4. **空态**：无计划时不占版面  
5. **同意条**（阶段 1）：规划就绪时「按此执行 / 修改计划 / 取消」  
6. **刷新/重连**：以 TurnView 最新 plan artifact 为准（须先修投影「最新覆盖」）

与活动条关系：活动条 = 瞬时；计划板 = 议程。两者都要，不能只留一个。

## 7. 风险与否决（保持严厉）

| 否决 | 原因 |
|------|------|
| 每 Turn 强制 Plan | 毁日常跟手 |
| **系统擅自切入 Plan 并执行** | 无模式门；比「建议」更严重 |
| 会话 sticky「连续推进」当 Plan | 忘关改节奏 |
| 话术自动进执行长跑 | 无契约 |
| 计划状态机十几种事件 | 信号过多 |
| 无同意门的多步主路径 | 假 Cursor |
| 未修「最新 plan」投影就做炫酷 UI | 进度说谎，比没有更糟 |
| 「建议切 Plan」刷屏 / 无法关闭 | 把辅路径做成骚扰 |

### 阶段 1 — Cursor 对齐（另立项）

| 票 | 内容 |
|----|------|

## 9. 决策摘要

1. **默认 = 普通 Agent**；Plan 是显式模式，不是默认人格。  
2. **入口：按钮为主，建议为辅**；建议可忽略，禁止静默变轨。  
3. **进 Plan ≠ 同意执行**；模式门与契约门分开。  
4. **清单是一等 UI**；步骤完成态必须在 Web 可见且为最新投影。  
5. **平台先于写作场景**；半成品补入口/面板/门闩，不新开引擎。  
6. 风险优先：假进度、强制 Plan、擅自切模式 —— 严重于「少一个连写」。

---

## 10. 开放问题

1. PlanPanel 放侧栏还是活动条下方（建议侧栏常驻 + 执行中活动条点出当前步标题）。  
2. status 枚举是否收敛为 `pending|in_progress|completed|cancelled` 四态写进合约。  
3. 模型漏更新 status 时，能否用「刚完成的 tool 名」做弱推断（易错，默认不做，优先教模型/`update_plan`）。  
4. 同意门用两 Turn 是否已满足自用，再开 B。
