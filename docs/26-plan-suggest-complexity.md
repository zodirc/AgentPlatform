# 26 — Plan 建议：复杂度判断（产品侧，不改交互）

> **状态：设计稿（待审）** · 2026-07-21  
> 前置：[`25-writing-runway`](25-writing-runway.md)（Plan 轨道 / 同意门已定）  
> 对照：[`13-rate-redlines`](13-rate-redlines.md) · [`10-product-experience`](10-product-experience.md) · ADR-014 · ADR-015  
> 约束：**不影响 Agent 速率；不改 Agent 交互逻辑**；只升级「何时冒泡建议切 Plan」

---

## 0. 一句话

在 **不碰 loop / 不碰同意门 / 不挡 TTFB** 的前提下，把今天「编号 ≥3 / 衔接词 ≥2」的窄启发式，升级成 **可解释的复杂度打分**，让建议条更像成熟产品（Cursor 式「复杂时才建议」），仍 **只建议、不自动进轨**。

本文 **不** 重开 Plan 模式设计；轨道 A/B、两道门、`plan_phase` 硬闸一律以 [`25`](25-writing-runway.md) 为准。

---

## 1. 问题

| 现状 | 体感缺口 |
|------|----------|
| UI `shouldSuggestPlanMode` 与 runtime `detect_plan_hint` 镜像同一套正则 | 只认「列表式多目标」，不认「多文件 / 难撤销 / 显式要方案」 |
| 无负向闸与会话冷却精修 | 短请求偶发误触；dismiss 后易再刷 |
| Intake 只塞软 `[plan_hint]` | 正确，但产品侧「为什么建议」对用户不可见 |
| 文档已禁止同步分类模型（25 §1.5 / R2） | 不能靠「再问一轮 LLM 该不该 Plan」凑成熟感 |

用户要的不是更强的自动变轨，而是：**该出现时出现、不该出现时安静**，且点进去之后的交互与今天完全一样。

---

## 2. 两条硬门槛（本文宪法）

### 2.1 不影响 Agent 速率（R1–R5）

| 红线 | 对本方案的含义 |
|------|----------------|
| **R1** | 建议计算不得推迟 `turn.accepted`；UI 侧在输入框本地算，不阻塞发送 |
| **R2** | **禁止**为「该不该建议 Plan」同步调模型（含小模型分类器、embedding 路由） |
| **R3** | 热路径只做毫秒级：正则 / 词表 / 轻量计数；禁止重 tokenizer、全库扫描 |
| **R4** | 语料标定、阈值调参、离线评测 → 异步 / CI；不上用户等待路径 |
| **R5** | 特征与阈值有单测；误触发 / 漏触发用表驱动用例，不靠感觉合入 |

### 2.2 不影响 Agent 交互逻辑

**冻结（本文零改动）：**

| 冻结项 | 说明 |
|--------|------|
| 默认 = 普通 Agent | 不因高分自动设 `plan_phase` |
| 进轨方式 | 仍仅：Plan 按钮 / 用户点「切换 Plan」 |
| 同意门 | 仍两 Turn：「规划」→「按此执行」；不做同 Run interrupt |
| 规划期 ToolScope | 仍硬闸；不因「建议」放宽或收紧 |
| 事件主路径 | 不新增必经节点；不改 `AgentEngine` while 语义 |
| Scenario 宪法 | 不在引擎内 `if scenario` 分支业务；差异只在 **评分权重表** |

**允许改的（仅建议面）：**

- 何时 `showPlanSuggest = true`
- 提示条文案（可带短 reason）
- 会话冷却 / dismiss 策略
- Intake `plan_hint` 文案是否带 reason（仍可选、不强制工具）

```text
优化面（本文）              冻结面（25）
┌─────────────────┐        ┌──────────────────────┐
│ 复杂度 score    │        │ Plan 按钮 / 相位      │
│ 建议条 + reason │ ─点头→ │ planning ToolScope    │
│ 冷却 / 负向闸   │        │ 按此执行 / executing  │
└─────────────────┘        │ PlanPanel / Cancel    │
                           └──────────────────────┘
```

## 3. 成熟 Agent 的合理做法（对照后采纳）

对照 Cursor Plan、常见编码 Agent 的 **交互抽像**（不抄 UI 皮）：

| 成熟做法 | 含义 | 我们采纳 |
|----------|------|----------|
| 按钮为主，建议为辅 | 用户永远能主动进 Plan | 已有；保持 |
| 复杂任务才建议 | 稀缺信号，不刷屏 | **本文升级触发质量** |
| 建议可忽略 | 不点 = 继续普通模式 | 已有；加强冷却 |
| 建议 ≠ 已进轨 | 不静默改模式 | **硬约束** |
| 进轨 ≠ 同意执行 | 先出清单再 Build | 已有（25） |
| 不靠同步分类模型猜模式 | 入口启发式 / 关键词 / 产品规则 | **硬约束**（ADR-014 / R2） |
| 默认同手短任务 | 改一句、问一句、继续 | 负向闸 |

**明确不抄 / 不做：**

- 无确认自动进 Plan 并开跑  
- 为建议再挂一轮同步 LLM  
- 把建议做成「后台已按 Plan 长跑」  
- 每个请求强制 Plan  
- 改同意门或 ToolScope 语义来「配合」更高召回  

成熟感来自 **触发准 + UX 少信号**，不是来自更强的自动操控。

## 4. 目标行为

### 4.1 输入 → 输出（纯建议面）

```text
message (+ 可选：scenario_id, 会话轻量态)
  → PlanSuggestDecision {
      suggest: bool,
      score: number,
      reasons: string[],   // 短、可展示、可测
      signals: string[], // 调试 / 单测用 id
    }
  → UI：suggest && !dismissed && !cooldown && !planMode → 提示条
  → Runtime（可选镜像）：score 达标 → metadata.plan_hint（软提示，不强制工具）
```

### 4.2 用户路径（与今天相同）

```text
轨道 A 输入复杂请求
  →（新）更准的提示条：「多文件改动风险高，建议先 Plan」+ [切换 Plan] [忽略]
  → 用户忽略：继续普通 Turn，行为与今天一致
  → 用户切换：plan_phase=planning …（25 既有）
```

### 4.3 成功标准（体验）

1. **跟手任务**（改语气、单点补丁、「继续」、短问）几乎不弹建议。  
2. **宽任务**（多目标 / 多章 / 多文件 / 「先给方案」）稳定出现建议，且带一句可读 reason。  
3. 同一会话 dismiss 或忽略后，在冷却窗内不再催。  
4. 打开建议 → 点切换 → 后续交互与未升级前 **比特级一致**（无新门、无新强制）。

## 5. 方案：可解释打分（确定性）

### 5.1 信号集（v1，全部毫秒级）

**正向（加分）：**

| id | 信号 | 示例 | 默认分 |
|----|------|------|--------|
| `multi_numbered` | 编号/列表目标 ≥3 | `1. … 2. … 3. …` | +3 |
| `multi_join` | 衔接词 ≥2 且长度 ≥40 | 然后/并且/also/finally | +2 |
| `explicit_plan` | 显式规划话术 | 先规划、给个方案再动手、分步做 | +3 |
| `multi_path` | `@` 路径或明确多文件 ≥2 | `@a @b`、第3章和第5章 | +2 |
| `high_risk_verb` | 高影响动词（窄词表） | 重构、重写全书、迁移、批量 | +2 |

**负向（减分 / 一票否决）：**

| id | 信号 | 示例 | 默认 |
|----|------|------|------|
| `too_short` | trim 后 &lt; 24 字 | 「改语气」 | score=0，不建议 |
| `continue_refine` | 继续/接着写/polish 精修 | 「继续」「再顺一下」 | −3 或否决 |
| `single_micro` | 单点微改词表 | typo、错字、改一句、语气 | −2 |
| `already_plan_prefix` | 已是 Plan/执行前缀消息 | `【Plan 模式】` | 否决 |
| `in_plan_mode` | 已在 planMode | — | UI 不展示（已有） |

**会话态（v1.5，仍不挡速率）：**

| id | 信号 | 说明 |
|----|------|------|
| `cooldown_active` | dismiss 后冷却 | 会话内或时窗（如 30min） |
| `recent_ignore` | 刚忽略建议又发短消息 | 压分 |
| `open_plan_stale` | 已有全 pending 计划却在普通模式狂发 | **可选**轻提示「你有未执行计划」，与「建议新建 Plan」文案区分；默认 v2 |

会话态只读 **前端已有 state** 或 **极轻 session 标记**；禁止为算分查库、扫 transcript 全文。

### 5.2 决策规则

```text
base = sum(signals)
若命中一票否决 → suggest=false
否则 suggest = (base ≥ threshold)
threshold 默认：writing=4, agent=4, interview=3（可表驱动）
reasons = 命中正向信号的人话列表（最多 2 条）
```

阈值与权重 **配置表 + 单测**，不写死在 prompt 里。

### 5.3 双端镜像纪律

| 端 | 职责 |
|----|------|
| **Web** | 唯一驱动提示条；打分实现为共享纯函数（TS） |
| **Runtime** | 可选镜像同一规则生成 `plan_hint`（软）；**不得**因 hint 改 ToolScope / 强制 `update_plan` |

规则变更必须 **两端同测例**（或抽共享 JSON 用例表），避免 UI 建议了、模型侧毫无弱提示（或反过来）。

### 5.4 文案（少信号）

- 主句固定短：`建议先切到 Plan，列出步骤再执行（可忽略）。`  
- 副句可选一行 reason：`检测到多个独立目标` / `请求涉及多处文稿` / `你提到先规划再动手`  
- **禁止** 恐吓式、禁止 列出内部 score

## 6. 明确非目标

| 非目标 | 理由 |
|--------|------|
| 同步 LLM / 小模型分类「该不该 Plan」 | R2、ADR-014、25 已否决 |
| 建议弹出即 `plan_phase=planning` | 静默变轨 |
| 提高建议召回而放宽规划期写工具 | 改交互逻辑 |
| 同 Run 同意 interrupt | 属 25 P2+，与本文无关 |
| 主模型异步 `suggest_plan` 事件（v1） | 可作 v2 辅信号；v1 先把确定性做准 |
| 用建议条替代 Plan 按钮 | 按钮仍是主入口 |

## 7. 与速率 / 交互的回归清单

合入前必须全部为真：

- [ ] 发送 Turn 的网络路径无新增 await（打分在客户端输入态完成）  
- [ ] runtime Intake 增加的工作 ≤ 一次正则扫描（与今天同量级）  
- [ ] 无新模型调用、无新 DB round-trip  
- [ ] 不点「切换 Plan」时：无 `plan_phase`、行为与旧版一致  
- [ ] 点「切换 Plan」后：planning / executing / 同意门与 25 一致  
- [ ] golden / 现有 plan 单测不因本文变红（除非显式更新建议用例）

## 8. 实施分层（票）

| 票 | 内容 | 速率/交互影响 | 状态 |
|----|------|----------------|------|
| **PS0** | 本文定稿；25 索引互链 | 无 | 设计稿 |
| **PS1** | 抽取 `PlanSuggestDecision`；正/负向信号表；TS↔Py 镜像；表驱动单测 | 无逻辑变轨 | 待做 |
| **PS2** | 提示条展示最多 1 条 reason；场景阈值表 | 仅文案 | 待做 |
| **PS3** | 会话冷却 + dismiss 精修（25 P3 收尾） | 仅建议频率 | 待做 |
| **PS4** | 离线/CI：建议语料集（应建议 / 不应建议）调阈值 | R4 | 待做 |
| **PS5**（可选） | 主模型异步提议事件，与规则 OR；仍可忽略 | 不挡 TTFB 才允许 | 不做除非 PS1–3 稳 |

**不做阶段：** 静默变轨、同步分类器、为建议改 ToolScope / 同意门。

## 10. 决策摘要

1. **只优化建议面**；Plan 交互与速率路径冻结在 25 / R1–R5。  
2. **成熟做法 = 稀缺、可忽略、可解释的建议**，不是更强的自动模式切换。  
3. **确定性打分 + 负向闸 + 冷却**；禁止同步 LLM 分类。  
4. **Web 驱动提示条，runtime 仅可选软 hint**；两端规则同测。  
5. 先 PS1–PS3 做准，再考虑异步模型辅信号（PS5）。
