# 26 — Plan 建议：复杂度判断（产品侧，不改交互）

> **状态：PS1–PS3 已落地；PS4 金标/调参脚手架已落地（提案不自动上线）** · 2026-07-21  
> 前置：[`25-writing-runway`](25-writing-runway.md)（Plan 轨道 / 同意门已定）  
> 对照：[`13-rate-redlines`](13-rate-redlines.md) · [`10-product-experience`](10-product-experience.md) · ADR-014 · ADR-015  
> **产品意图（硬）：**  
> 1. **尽量不影响** Agent 交互速率与 Agent 交互逻辑  
> 2. **尽量偏向** 成熟 Agent 的入口思路（建议稀缺、可忽略、人确认后才进轨）  
>
> 实现：`services/web/.../planSuggest.ts` ↔ `services/runtime/.../plan_suggest.py`；用例 `eval/plan_suggest/cases.json`

---

## 0. 一句话

在 **不碰 loop / 不碰同意门 / 不挡 TTFB** 的前提下，用 **可解释的复杂度打分** 决定「要不要冒泡建议切 Plan」——对齐 Cursor 式「复杂时才建议」，仍 **只建议、不自动进轨**。

本文 **不** 重开 Plan 模式设计；轨道 A/B、两道门、`plan_phase` 硬闸一律以 [`25`](25-writing-runway.md) 为准。

**成熟度自检（当前）：** 入口策略成熟（建议→确认→执行）；触发器合格可用（规则打分）；判断力半成熟（字面启发式，尚未语料标定）。增强判断力 **只加厚建议面**，不换交互哲学。

---

## 1. 问题与意图

### 1.1 用户要什么

不是更强的自动变轨，而是：

- **该出现时出现、不该出现时安静**
- 点进 Plan 之后的交互与未建议时 **完全一样**
- 体感靠近成熟产品：按钮为主、建议为辅、可忽略、不刷屏

### 1.2 已落地（PS1–PS3）解决了什么

| 以前 | 现在 |
|------|------|
| 几乎只认「1.2.3.」或衔接词 | 显式要方案、多文件、高影响动词也可加分 |
| 短改偶发误触；忽略后易再弹 | 负向闸 + 30min 冷却 |
| 建议条无原因 | 最多一行可读 reason |
| Web / runtime 启发式易分叉 | 共享 `cases.json` 表驱动对齐 |

### 1.3 仍缺什么（判断力）

- 阈值 / 权重多靠验收用例拍齐，**未用真实对话标定**
- 信号仍偏「撞词 / 撞列表」，对「平淡措辞的大任务」易漏报
- 会话态（越改越大、连续受挫）几乎未用
- TS / Py 双份规则，长期有漂移风险

## 2. 两条硬门槛（本文宪法）

> 任何「增强判断力」的提案，若触碰下列红线，直接否决。

### 2.1 尽量不影响 Agent 速率（R1–R5）

| 红线 | 对本方案的含义 |
|------|----------------|
| **R1** | 建议计算不得推迟 `turn.accepted`；UI 在输入框本地算，不阻塞发送 |
| **R2** | **禁止**为「该不该建议 Plan」**同步**调模型（含小模型分类器、embedding 路由） |
| **R3** | 热路径只做毫秒级：正则 / 词表 / 轻量计数；禁止重 tokenizer、全库扫描 |
| **R4** | 语料标定、阈值调参、离线评测 → 异步 / CI；不上用户等待路径 |
| **R5** | 特征与阈值有单测；误触发 / 漏触发用表驱动用例合入 |

### 2.2 尽量不影响 Agent 交互逻辑

**冻结（增强判断力时也零改动）：**

| 冻结项 | 说明 |
|--------|------|
| 默认 = 普通 Agent | 不因高分自动设 `plan_phase` |
| 进轨方式 | 仍仅：Plan 按钮 / 用户点「切换 Plan」 |
| 同意门 | 仍两 Turn：「规划」→「按此执行」 |
| 规划期 ToolScope | 仍硬闸；不因「建议」放宽或收紧 |
| 事件主路径 / loop | 不新增必经节点；不改 `AgentEngine` while |
| Scenario 宪法 | 不在引擎内 `if scenario` 分支业务；差异只在 **评分权重表** |

**允许改的（仅建议面）：**

- 何时 `showPlanSuggest = true`、reason 文案、冷却
- 打分特征 / 阈值 / 词表（离线标定后合入）
- Intake `plan_hint` 文案（软；不强制工具）
- 可选：主模型 **异步**「建议切 Plan」事件（与规则 OR，可忽略，不挡 TTFB）

```text
优化面（本文）              冻结面（25）
┌─────────────────┐        ┌──────────────────────┐
│ 复杂度 score    │        │ Plan 按钮 / 相位      │
│ 建议条 + reason │ ─点头→ │ planning ToolScope    │
│ 冷却 / 负向闸   │        │ 按此执行 / executing  │
│ 判断力加厚 ↓    │        │ PlanPanel / Cancel    │
└─────────────────┘        └──────────────────────┘
```

## 3. 成熟 Agent 对照（我们偏向什么）

对照 Cursor Plan 等产品的 **交互抽像**（不抄 UI）：

| 成熟做法 | 含义 | 我们 |
|----------|------|------|
| 按钮为主，建议为辅 | 用户永远能主动进 Plan | ✅ 保持 |
| 复杂任务才建议 | 稀缺，不刷屏 | ✅ 阈值偏稀缺；继续用标定收紧/放宽 |
| 建议可忽略 | 不点 = 普通模式 | ✅ + 冷却 |
| 建议 ≠ 已进轨 | 不静默改模式 | ✅ 硬约束 |
| 进轨 ≠ 同意执行 | 先出清单再 Build | ✅ 见 25 |
| 不靠同步分类猜模式 | 规则 / 关键词 / 产品启发式 | ✅ R2 |
| 「Agent 觉得」可提议 | 跑着时可建议切 Plan，人决定 | ⏳ PS5：异步辅信号，非闸门 |
| 默认同手短任务 | 改一句、继续、短问 | ✅ 负向闸；PS4b 加厚 |

**明确不抄：**

- 无确认自动进 Plan 并开跑  
- 每条消息同步 LLM「该不该 Plan」  
- 建议条背后已按 Plan 长跑  
- 用放宽 ToolScope / 改同意门换召回  

**成熟感公式：** `触发准 + 少骚扰 + 人控门` ≫ `更强的自动操控`。

## 4. 当前机制（人话 + 规格）

### 4.1 人话

把用户输入打成一个分数；**够线只弹建议条，不够就安静**。  
**到线 ≠ 必须 Plan**，更不是自动进轨。

### 4.2 阈值（已落地）

| 场景 | 建议线 |
|------|--------|
| `writing` | **≥ 4** |
| `agent` | **≥ 4** |
| `interview` | **≥ 3** |

### 4.3 信号集（v1，已落地）

**正向（加分）：**

| id | 信号 | 示例 | 默认分 |
|----|------|------|--------|
| `multi_numbered` | 编号/列表目标 ≥3 | `1. … 2. … 3. …` | +4 |
| `multi_join` | 衔接词 ≥2 且长度 ≥40 | 然后/并且/also/finally | +2 |
| `explicit_plan` | 显式规划话术 | 先规划、给个方案再动手 | +4 |
| `multi_path` | `@` 路径或章节 ≥2 | `@a @b`、第3章和第5章 | +2 |
| `high_risk_verb` | 高影响词（可叠） | 重构、迁移… | +2 / 词，上限 +4 |

**负向（减分 / 否决）：**

| id | 信号 | 示例 | 默认 |
|----|------|------|------|
| `too_short` | 过短且无强信号 | 「改语气」 | 否决 |
| `continue_refine` | 继续/精修 | 「继续」 | 否决或 −3 |
| `single_micro` | 微改词表 | typo、错字、语气 | −2 |
| `already_plan_prefix` | 已是 Plan 前缀消息 | `【Plan 模式】` | 否决 |
| `cooldown_active` | dismiss 后 30min | — | 否决（UI） |

### 4.4 数据流（不变）

```text
message (+ scenario_id, cooldown?)
  → PlanSuggestDecision { suggest, score, reasons, signals }
  → UI：suggest && !planMode && !busy && !cooldown → 提示条（+ 一行 reason）
  → Runtime：suggest → 软 plan_hint（不强制工具、不设 plan_phase）
```

## 5. 如何增强判断力（最新路线）

> 目标：从「合格启发式」走到「更像成熟产品的稀缺建议」。  
> 手段：只加厚建议面；**不**换门闩、**不**同步分类模型。

### 5.1 总序（按性价比）

```text
① PS4   真实语料标定阈值/权重     ← 判断力主杠杆
② PS4b  负向闸 + 场景分表加厚     ← 少误伤（更成熟）
③ PS4c  会话态轻信号               ← 少漏报（仍本地/已有 state）
④ PS4d  规则单源（JSON/YAML）      ← 防 TS↔Py 漂移
⑤ PS5   主模型异步「建议切 Plan」  ← 成熟「Agent 觉得」；非闸门
```

任一票合入前仍过 §2 与 §7。

### 5.2 PS4 — 语料标定（优先）

| 项 | 约定 |
|----|------|
| 目的 | 让「≥4 分」来自数据，而不是拍脑袋 |
| 做法 | `eval/plan_suggest/golden.jsonl` 标注 `should_suggest`；`scripts/plan_suggest_eval.py` 跑基线；`--tune` 网格搜索写出提案 |
| 自动调权？ | **会搜索并写 `reports/latest_tune.json`，不会静默改 `weights.json`**。人工覆盖配置后再重建 |
| 指标 | 精确率优先于召回（默认 tune 约束 P≥0.85） |
| 速率/交互 | **零**：纯离线（R4） |
| 命令 | `make eval-plan-suggest` / `make eval-plan-suggest-tune` |
| 产出 | 金标 + 报告；**上线权重只改** `packages/contracts/plan_suggest/weights.json` |

金标可继续追加；`cases.json` 仍是 CI 快测子集。

### 5.3 PS4b — 负向与场景分表（少误伤）

成熟判断力一半在 **不弹**。

| 方向 | 例 | 约束 |
|------|----|------|
| writing 跟手负向 | 「写第三章」「改这段」「什么意思」单独出现 → 压分 | 若同句已有大纲+引用+多章，仍可加分 |
| agent 跟手负向 | 单文件 typo、短问「这个函数干嘛的」→ 压分 | |
| 场景阈值/词表分表 | writing vs agent 的 `high_risk` / 章节规则不同 | 仍只在权重表，不进引擎分支 |
| 结构信号（轻） | 多个动词短语、多个「和/以及」连接的作业对象 | 毫秒级；禁止 NLP 重模型 |

### 5.4 PS4c — 会话态（少漏报，仍不挡速率）

只读 **前端已有 state** 或极轻标记；**禁止**为算分查库、扫全文 transcript。

| 信号 | 含义 | 建议行为 |
|------|------|----------|
| `cooldown_active` | 已落地 | 保持 |
| `recent_ignore` | 刚忽略建议又发短消息 | 压分 |
| `streak_friction` | 同会话连续大 patch 被拒 / 工具失败（若 UI 已有计数） | 小幅加分 |
| `open_plan_stale` | 已有全 pending 计划却在普通模式狂发 | **另文案**「你有未执行计划」，与「建议新建 Plan」区分；勿混成一条 |

### 5.5 PS4d — 规则单源（已落地）

**唯一配置：** [`packages/contracts/plan_suggest/weights.json`](../packages/contracts/plan_suggest/weights.json)

- Web / Runtime / eval 都读这一份（分数、阈值、高风险词、reason、冷却）。  
- 正则仍在代码里（改规则才动 TS/Py）。  
- 改灵敏度：编辑 JSON → `make eval-plan-suggest` → 重建 web + runtime。  
- tune 的 `proposed_config` 可整份覆盖该文件（人工确认后）。

### 5.6 PS5 — 主模型异步提议（成熟「Agent 觉得」）

| 项 | 约定 |
|----|------|
| 是什么 | 普通 Turn 进行中，模型可发出可忽略的「建议切 Plan」信号（事件或轻量字段） |
| 不是什么 | **不是** Intake 同步分类；**不是**自动 `plan_phase=planning` |
| 与规则关系 | 与确定性打分 **OR**：任一为真可出条；人点头才进轨 |
| 速率 | 必须不挡 TTFB；信号出现在 stream 途中或 turn 末，用户已在读回复 |
| 前置 | PS4 标定稳定后再做；避免「模型乱建议」叠加未校准规则导致刷屏 |

### 5.7 明确仍不做（增强判断力时也禁止）

| 做法 | 为何否决 |
|------|----------|
| 每条消息同步 LLM / 小模型「该不该 Plan」 | 伤速率（R2）；也不比好规则更「成熟」 |
| 分数一高就自动进 Plan | 伤交互逻辑；假成熟 |
| 为提高召回放宽规划期写工具 | 改 25 门闩 |
| 用建议条替代 Plan 按钮 | 主入口必须仍是显式控件 |

## 6. 目标行为与成功标准

### 6.1 用户路径（与 25 相同）

```text
轨道 A 输入
  →（建议面）可能出现提示条 + reason + [切换 Plan] [忽略]
  → 忽略：继续普通 Turn（逻辑不变）
  → 切换：plan_phase=planning …（25 既有）
```

### 6.2 体验成功标准

1. 跟手任务几乎不弹。  
2. 宽任务稳定弹，且有一句可读 reason。  
3. 忽略后冷却窗内不催。  
4. 点「切换 Plan」后交互与未升级建议前 **比特级一致**。  
5. **（PS4 后）** 金标集上精确率达标（具体阈值写入调参记录，默认偏稀缺）。

## 10. 决策摘要

1. **产品意图：** 不伤速率、不改交互逻辑；建议面尽量成熟（稀缺、可忽略、人控门）。  
2. **已落地：** 可解释打分 + reason + 冷却；到线只建议，不自动 Plan。  
3. **判断力主路径：** 语料标定（PS4 脚手架已落地）→ 负向/场景（PS4b）→ …；**tune 只出提案，人工合入权重。**  
4. **禁止：** 同步分类、静默变轨、用改门闩换召回。  
5. **成熟感来自触发准与少骚扰，不来自更强自动操控。**
