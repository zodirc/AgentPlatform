# 24 — 写作 Token 经济（自然交互优先）

> **状态：WT1–WT4 已落地（2026-07-20）；WT5（prompt cache 布局）设计已定、实现延后** · 实现随 writing monofile / work index  
> 前置：[`23-writing-work-model`](23-writing-work-model.md)（作品树 / monofile 已落地）  
> 对照：[`13-rate-redlines`](13-rate-redlines.md) · [`20-context-compaction-walkthrough`](20-context-compaction-walkthrough.md) · [`14-writing-quality`](14-writing-quality.md) · [`10-product-experience`](10-product-experience.md)  
> 约束：**不伤交互、不伤速率、不为省钱牺牲成稿效果**；**不为 cache 命中率拆墙**

---

## 1. 问题

长写作（优化大纲 + 写数章）费用高，主因通常不是「模型贵一点」，而是：

| 成本层 | 典型来源 | 是否必要 |
|--------|----------|----------|
| **输出** | 章正文、大纲全文 | **必要**（省这里 = 少写） |
| **输入 · 对话窗** | 同会话 transcript 越滚越长（工具结果、旧 diff、旧读文件） | 大半可压 |
| **输入 · 误读全书** | 续写第 N 章时把整本 `manuscript.md` 读进上下文 | 常可避免 |
| **输入 · 固定开销** | system + tools + work index + cards | 应有硬顶，已部分控制 |

[`23`](23-writing-work-model.md) 已解决「换会话拆书」；**本文只谈：在书不断的前提下，如何自然、成熟地控输入成本。**

**明确非目标：** 压缩生成长度、降采样写稿质量、每轮强制再调一个 LLM「省钱裁判」。

---

## 2. 设计原则（三条硬门槛）

### 2.1 不影响 Agent 交互与速率（R1–R5）

| 红线 | 对本方案的含义 |
|------|----------------|
| **R1** | 任何装载 / 书签逻辑不得推迟 `turn.accepted` |
| **R2** | **禁止**为首 token 再同步调模型（含「先摘要再写」） |
| **R3** | 热路径只做毫秒级：抽章节块、拼短书签、截断已知超长 tool 结果 |
| **R4** | 重摘要、全稿扫描、质量评分 → 异步 / 用户触发（`/compact`） |
| **R5** | 有单测或 golden；延迟相关要可观测（`context.reported`） |

交互上：**用户仍说人话**（「写第三章」「接着写」），不要求先选章节下拉、不强制先 `/compact`。系统静默做「该带什么进窗」。

### 2.2 成熟、自然（Mature agent 体验）

成熟做法是 **状态在磁盘、对话在记忆、窗口按任务切片**，而不是：

- 教用户「为了省钱请新开会话」（书已能跨会话，但不应是唯一手段）
- 暴露 token 仪表盘当主交互
- 用一堆 slash 替代自然语言

自然路径：

```text
用户：写第三章
  → 系统已知 manuscript / work index
  → 本轮默认只装：短索引 + 第2章尾（可选）+ 第3章现有块（若有）
  → 模型 draft_section(ch3) / patch
  → 长会话后：用户说「压缩一下」或 UI 轻提示 /compact（带写作书签）
```

### 2.3 效果优先：不能得不偿失

省 token 若导致：人物前后矛盾、章衔接断裂、漏改大纲、乱引用资料——就是失败。

**效果护栏（必须保留的上下文）：**

| 保留 | 理由 |
|------|------|
| 当前章全文（在编块） | 续写 / 改写的作业面 |
| 上一章尾（有硬顶，如末 1–2k 字） | 衔接与伏笔，不是全书 |
| 短 work index + outline TOC | 定位「写到哪」 |
| 与本轮相关的 writing cards（已有预算） | 人设/文风；不可为省钱整类丢掉 |
| 用户本轮明确 `@` 的文件 | 用户意图优先于自动裁剪 |

**可以砍的：**

| 可砍 | 条件 |
|------|------|
| 更早章节的全文 | 除非用户点名或 `@` |
| 过期 tool 轨迹 | 已有 collapse/snip；写作可更早 microcompact |
| 重复读同一大文件 | 同 turn 内命中缓存 / 指针 |
| 无焦点的「通读 manuscript」 | 默认拒或改为抽块工具 |

**否决的「省钱」手段（得不偿失）：**

1. 每轮自动 LLM 摘要再开写（R2 + 双倍延迟 + 摘要丢细节）  
2. 强制把成稿章截断进窗当「代表」（丢伏笔）  
3. 关掉 cards / RAG 来省固定开销（WQ 回退）  
4. 自动 `/compact` 且不带写作书签（丢「写到哪」）  
5. 限制 `draft_section` 输出长度（直接伤成稿）  
6. 为省输入禁止 `read_file`（模型瞎编衔接）

## 3. 现状（已有，勿重复造轮）

| 能力 | 作用 | 缺口 |
|------|------|------|
| 作品树 + monofile | 换会话书还在 | 不直接降本轮输入 |
| Work index（短、无 LLM） | 启动定位 | 不装章节正文 |
| 窗内 compaction（budget→…→autocompact） | 防爆窗 | 不理解「当前章」 |
| `/compact` | 砍 session transcript | 摘要未必带写作书签 |
| tool 结果字符预算 | 砍单条工具输出 | 写作读稿仍可能一次过大 |
| cards / project_context 硬顶 | 控固定开销 | OK，勿再加同步 LLM |

## 4. 推荐方案（分层）

整体策略：**磁盘是全书；模型窗是「当前作业面」。**

### 4.1 WT1 — 按章作业面装载（核心，自然且对效果友好）

**做什么**

- 从用户话术 / 最近 `draft_section` / manifest 推断 `focus_section_id`（启发式，**无 LLM**）
- Turn 启动或首次需要正文时，提供 **章节抽块**（复用 monofile `<!-- section:id -->`）：
  - `focus` 全块（可配上限，超限则头+尾+中间省略标记，并提示可分段改）
  - `prev` 仅尾部 N 字符（默认 ~1500–2500）
- **默认不**把整本 `manuscript.md` 注入 system；`read_file(manuscript.md)` 在 writing 下可改为：
  - 返回「请改用抽块 / 指定 section」+ 目录，或  
  - 自动只返回 focus+prev（工具结果里说明裁剪原因）

**交互**

- 用户仍说「写第三章」「接着写」——零新 UI 门槛  
- 要通读：用户明确说「通读全文 / 全文检查」或 `@manuscript.md` 且 intent=review → 允许更大窗或分段工具

**速率**

- 纯文件抽块，R1–R3 友好；禁止为此调模型  

**效果**

- **增益：** 衔接靠 prev 尾，正文作业面完整 → 通常优于「带着十章噪声」  
- **风险：** focus 识别错章 → 缓解：work index 列出已有 section；错了用户一句纠正即可；识别失败则退回「只带 index、让模型 `read` 抽块」  
- **风险：** 大章超上限被腰斩 → 必须可见省略标记，且允许「只改上半/下半」；禁止静默丢段

### 4.2 WT2 — Writing 感知的 `/compact` 书签（同会话省钱主力）

**做什么**

- `/compact`（及 UI「压缩会话」）在写作场景生成摘要时 **强制带书签结构**（模板，确定性拼接 + 可选短 LLM 摘要）：

```text
[writing bookmark]
manuscript: manuscript.md | draft: .agent/work/drafts/manuscript.md
focus: ch3
sections_present: ch1, ch2, ch3
outline: <TOC 截断>
notes: <既有摘要 / 用户目标一句>
```

- 压缩后 transcript 替换为书签 + 摘要；**不删磁盘文稿**
- 下一 turn 自动把书签拼进 runtime/system 旁路（短、有硬顶）

**交互**

- 保持现有 slash；可在长会话 UI 轻提示「对话变长，压缩可省用量且不丢书」，**不自动强制**（避免打断心流；自动压缩仅保留窗内链）

**速率**

- 用户触发；可用已有 `compact_model_*` 便宜模型；不挡 TTFB  

**效果**

- **增益：** 砍掉工具史，保留「写到哪」→ 续写不漂  
- **风险：** 摘要丢掉用户刚说的细改要求 → 缓解：书签外保留「最近 1 条 user 原文」或「最近 K 轮 user」不进 LLM 摘要直接粘贴（确定性）

### 4.3 WT3 — 写作工具结果更「对症」的截断（低风险）

**截的是读回上下文的噪音，不是成稿篇幅。**

| 会截 / 降级 | 不会截 |
|-------------|--------|
| `read_file(manuscript)` 整本灌窗 | `draft_section` / `propose_patch` 写出多长都行 |
| 普通超长 tool_result（4k budget） | 带 `writing_section_extract` 的章节抽块（免腰斩） |
| 同 turn 重复相同 read（既有 cache `_note`） | 工具错误信息 |

长章用 **多个动作**（分段 patch / 多轮 draft），与省 token 同向。

### 4.4 WT4 — 产品引导的阶段拆分（零引擎侵入）

**做什么**

- 文案 / 空态：建议「大纲一轮 → 一章一轮或一章一压缩点」  
- **不**强制新 Session；强调 `/compact` 与新会话等价于「换对话、不换书」

**效果：** 行为省钱，无技术风险。  
**风险：** 无；勿做成强制向导。

### 4.5 明确延后 / 不做

| 项 | 原因 |
|----|------|
| 每 turn 自动 LLM compact | R2、延迟、细节丢失 |
| 按 token 计费打断生成 | 伤交互与成稿 |
| 向量化全书再检索当上下文 | 重、延迟、写作连贯性差；RAG 留给资料库 |
| 多作品 Work 表 | 与 token 无关；见 23§11 |
| 为刷 prompt-cache 命中率牺牲上下文 | 见 §10；得不偿失 |

### 4.6 WT5 — Prompt cache 布局（待落地 · 不拆墙）

**观察（2026-07）：** 仪表盘上写作会话常见 **~10–15% prompt cache hit**。主因往往不是「系统坏了」，而是计费输入里大半是 **逐步增长的 messages / tool_results**——本来就不会进稳定前缀 cache。把命中率刷高却多灌上下文，是拆东墙补西墙。

**原则（三条）：**

1. **先降绝对未命中输入**（WT1–WT4 同向）：少整本读、工具结果硬顶、按章作业面 → miss 分母变小，账单直接降；命中率可能几乎不动，仍算成功。  
2. **再理顺稳定前缀**（同内容重组，不删护栏）：见下表。  
3. **禁止用质量 / 跟手 / 交互换命中率数字。**

**现状组装（问题）：** `prepare_writing_system_prompt` 把 cards + work index + focus/prev **焊进整段 system**，再整块打 cache。换用户话 / 换章 → **整段 system 一起 miss**，连稳定的 `system.md` 正文也陪葬。

**推荐布局（内容不减，只改位置）：**

| 层 | 放什么 | cache |
|----|--------|--------|
| **稳定** | `system.md` 正文 + tools | 打标；跨 Turn 尽量命中 |
| **易变** | cards / focus / prev / work index | 放 runtime 或靠后 user；**照样送给模型** |

**行为上少制造「肥 Turn」（零引擎也能做）：** 改名走 `rename_file`、少无谓 `delegate`、少为格式校验整本拆读。少步数 = 少次全量重送。

**明确不要做（拆墙清单）：**

| 做法 | 为什么否决 |
|------|------------|
| 为 cache 丢掉 focus / prev / cards | 质量换命中 |
| 每步自动 LLM 摘要再写 | 伤 R2，还可能多一轮费 |
| 强逼短输出 / 少写章 | 省输出毁产品 |
| 死磕命中率仪表盘 | 可刷高命中、总费用不降甚至升 |
| 更勤的 collapse 专为 cache | 易丢细节，且改历史会再 miss |

**验收：** 同任务「写一章」的 **计费 input / cache miss 绝对量**下降；质量与首 token 不回归。**不**以「命中率 %」为唯一 KPI。

**状态：** 设计已定，实现延后（与 WT1–WT4 解耦；先确认按章装载在实战里生效，再动组装）。

## 5. 效果评估（合并前必须想清楚）

### 5.1 质量信号（宁可少省，不可瞎省）

落地后用写作 golden / 手工剧本看：

1. **衔接：** 第 N 章开头是否接得上 N−1 尾（有 prev 尾 vs 无）  
2. **一致：** 人设/称谓是否因丢掉 cards 而漂（cards 预算不得为 WT 让路到 0）  
3. **定位：** compact 后是否还知道 focus 章  
4. **误伤：** 用户要「全文统一人称」时，系统是否仍允许扩大装载

### 5.2 成本信号

- `context.reported`：同任务「写一章」的 `tokens_before` 中位下降  
- 不要求输出 token 下降（正文长度由用户决定）  
- Prompt cache（WT5）：看 **cache miss / 总 input 绝对量**；命中率仅作辅证，不作唯一目标

### 5.3 速率信号

- 首 token / `turn.accepted` 无回归（对照 13）  
- 抽块 P99 ≪ 模型 RTT

**合并准则：** 成本降 + 质量不降 + 速率不降 → 合；任一质量红灯 → 回退该开关。

## 6. 与用户心智的对齐

| 用户感觉 | 系统实际 |
|----------|----------|
| 我在写一本书 | `manuscript.md` 在磁盘变长 |
| 我在和一个助手聊天 | Session transcript；可变长 |
| 聊太久有点贵 | `/compact` 或新会话；书还在 |
| 接着写第三章 | 窗里主要是第三章作业面，不是一至十章全文 |

省 token = **缩小聊天窗与误读**，不是 **缩短小说**。

## 8. 决策摘要

1. Token 经济作用于 **输入窗与误读**，不作用于成稿长度。  
2. 成熟形态 = **磁盘全书 + 按章作业面 + 带书签的会话压缩**。  
3. 一切热路径无同步 LLM；遵守 R1–R5。  
4. 效果护栏：focus 全文（或显式分段）、prev 尾、cards、用户 `@` / 全文意图。  
5. 拒绝自动摘要开写、砍质量能力、无书签 compact。  
6. Prompt cache：**先砍绝对 miss，再拆稳定/易变前缀**；不为命中率拆墙（§10 / WT5）。

---

## 9. 开放问题

1. focus 识别：仅规则 vs 极轻量分类（仍避免同步 LLM；规则优先）。  
2. prev 尾默认字符：1500 vs 2500（用衔接盲测定）。  
3. `read_file(manuscript.md)`：硬拒绝整文件 vs 自动抽 focus——建议 **自动抽 + 说明**，全文意图白名单放行。  
4. 是否在 Usage 旁显示「本轮约带了 chN」——增强信任，非必须。  
5. WT5：易变块挂 `runtime_context` 还是独立 user message；与现有 `cache_control` / fingerprint 如何对齐（实现时再定）。

## 10. Prompt cache（决策备忘 · 2026-07）

> 与 §4.6 WT5 同文；此处便于检索「命中率该不该优化」。

**合理路径 = 少送废话（绝对输入）+ 稳定段别被本轮作业面污染（同内容重组）。**  
前者降账单，后者抬一点真命中；两者都不牺牲写作效果和交互。

落地顺序建议：先确认 WT1–WT4 在实战会话里真的在跑 → 再开 WT5 组装票。
