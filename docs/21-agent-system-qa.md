## 一句话内核

不是「固定流水线」（先分类意图再挂死节点），而是 **单循环 + 工具中心 + 流式事件 + 上下文压缩**：

```text
用户发消息
  → 控制面建一轮对话；执行面先回执、再干活
  → 确定性预处理（少数命令可零模型短路；多目标可弱提示）
  → while 未超步数 / 预算:
        组装上下文 + 本步工具表
        流式调用模型（Function Calling）
        有 tool_calls → 校验 / 审批 / 执行 → 结果写回
        无 → 本轮结束
  → 事件入库，由控制面推到界面；索引 / 评价等重活走旁路
```

| 分工 | 谁负责 | 为什么 |
|------|--------|--------|
| **意图** | 模型：选工具、填参数、写正文 | 开放路径画不全；画死就空跑 |
| **事实** | 运行时：校验、沙箱、检索结果、引用核对、超时取消 | 概率模型不能当真相源 |
| **看见什么** | Context engineering | 长会话必膨胀；不治理则延迟、费用、注意力一起崩 |
| **厚度** | Harness | 「能聊天」≠「能交付」；**加厚不许挡首 token** |

三条总纲（后文反复出现）：

1. **messages 即状态** — 轨迹在有序消息里，不靠膨胀的业务状态对象。  
2. **加能力 = 加工具 / 加厚，不是加节点** — 冻循环内核，用场景配置扩工具面、提示、审批与布局。  
3. **热路径只放必要工作** — 建索引、打分、大批量核查进旁路；正确性可短暂降级，延迟优先。

部署直觉：边缘网关 → 前端；控制面（鉴权、建轮、独占推送）↔ 执行面（跑循环）↔ 工作区与数据库。前端只消费事件投影，不猜阶段。

## 术语速查

| 词 | 一句话 |
|----|--------|
| **一轮对话 / 一次运行 / 一步** | 用户一条消息 → 执行实例（1:1）→ 循环内「组装→模型→工具」；取消后不可恢复同一运行 |
| **热路径 / 旁路** | 用户必须等的路径 vs 不挡交互的异步/预热/离线活 |
| **工具中介** | 资料与记忆不预灌；模型调工具后才进窗 |
| **证据集 / 未核验** | 本轮捞到的引用 id vs 写了引用但本轮没捞到（标脏，默认不硬阻断） |
| **素材钉选** | 人设/文风卡钉进系统侧；**不进 RAG**，防风格句假命中 |
| **文件指针** | 委派时只传路径，不传父对话全文；需要再读盘 |
| **填充率** | 待发内容估算 token ÷ 可用容量；触发压缩阶梯（见 Q9） |
| **会话 / 作品** | 对话单位 vs 书稿真源；换会话不拆书（见 Q23） |
| **Golden / Rubric / SLO** | 行为回归用例 / 离线质量启发式 / 体验延迟红线 |

**速率五条**（与效果冲突时，交互与速率优先）：  
① 不推迟「已受理」；② 首 token 前不加同步裁判模型；③ 热路径 CPU 毫秒级；④ 重活异步/离线/用户触发；⑤ 可测才合并。

## Q0. 落地场景是什么？是否有用？

**结论：✅ 写作为主，沙箱 Agent 为辅；检索 → 起草 → 引用核查闭环已通。**

| 场景 | 意味着 |
|------|--------|
| 写作 | 检索、素材钉选、分节起草、可审改稿、导出、确定性引用核查 |
| 沙箱 Agent | 更宽工具面；写/执行走审批；记忆按需；多表查询仍为空壳 |

**「有用」四字**：跟手、可控、有据、可回归——不是「偶尔写得漂亮」。

**边界要主动讲**：成文质量仍靠线上模型；无企业多租户 ACL；多表未接真表；prompt 稳定/易变分家利于缓存，设计已定、布局按排期。

## Q1. 幻觉或乱调工具如何缓解？

**结论：🔸 幻觉灭不掉；用入口与证据加压，而不是再问模型「你确定吗」。**

再调模型自检：多付延迟、仍可能自信地错、把不可复现判断当门禁。正确策略是 **缩小动作空间 + 确定性门 + 可核对痕迹**。

| 层 | 做法 |
|----|------|
| 动作空间 | 场景白名单；后期可再拿掉检索/委派/记忆，减空转 |
| 入口 | 参数不合 schema → 结构化错误回灌，**不执行** |
| 证据 | 本轮证据集；草稿引用不在集 → 标未核验（默认不硬阻断） |
| 核对 | 用户触发的零模型磁盘扫描（锚点在不在）；覆盖靠离线抽样，不挂每次交付尾 |
| 观测 | 误用 / schema 失败可计数 |

**两种核对不要混**：

| | 本轮证据集 | 事实核查扫描 |
|--|------------|----------------|
| 问的是 | 这轮有没有检索到该 id | 磁盘上引用能不能指到文件 |
| 调模型吗 | 否（记账） | 否（入口可短路） |
| 能证明史学正确吗 | 不能 | 不能 |

零命中时「别编引用」仍部分靠模型守提示——这是诚实边界。

## Q2. 怎么评价 Agent 好坏？

**结论：✅ 分层评价；内容质量分永不挂用户本轮尾。**

| 层 | 测什么 | 何时 |
|----|--------|------|
| 行为契约 | 该调的工具、格式 | CI / 本地 |
| 体验 SLO | 受理、首 token、取消时延 | 在线 |
| 工具健康 | 误用、schema 失败 | 在线 |
| 检索 | hit、索引滞后 | 离线/专项 |
| 内容质量 | 启发式抽样 | **离线** |
| 交付诚实 | 完成是否被误当成正确；导出范围是否显式 | 产品纪律（Q25） |

在线证明跟手与可控；「写得好不好」离线抽。同步裁判模型挂轮末 = 用用户等待买评价。

## Q3. 任务规划如何保证拆解合理？

**结论：🔸 规划可选，不是强制阶段。系统给弱提示、跨轮备忘、可选 Plan；不替模型先开计划会。**

### 为什么不加强制门？

「改短第三节」也要先付一整轮规划调用 → 用户发呆、无正文 → 违背热路径纪律。  
**简单可零规划；复杂可规划，不强制。**

### 默认轨道（不进 Plan 也能活）

1. **弱提示**：多编号目标 / 多承接词 → 塞一行「可先更新计划」；模型可无视。  
2. **可选计划工具**：待办写进对话，不是单独编排节点。  
3. **跨轮未完成项**：结束后抽短列表进会话摘要，方便「继续」。

### Plan 模式：两道门

```text
默认普通 Agent（跟手）
  → 用户点 Plan，或点「建议先计划」（禁止静默变轨）
  → 规划期：工具面收窄偏只读 → 产出步骤清单
  → 用户「按此执行」← 第二道门
  → 执行期：按项推进，清单状态可见；可随时停
```

进 Plan ≠ 同意执行。规划期硬收窄可写工具，避免「口头在规划、手里已改文件」。

### 「建议切 Plan」怎么判？

毫秒级可解释打分：多目标 / 显式要方案 / 多文件 / 高风险动词加分；过短、微改、「继续」减分。过阈值才冒泡；忽略后冷却，防刷屏。

**禁止**为「该不该 Plan」再同步调分类模型。权重可离线提案，**不**自动改线上。判断力仍偏字面——平淡措辞的大任务可能漏报，要主动讲。

### 「合理」保证吗？

不能数学保证。只能提高先拆概率、让拆解可见、用步数预算防无限拆。漏步靠提示、打断与事后核查。

## Q4. 子 Agent 上下文怎么传？

**结论：✅ 任务说明 + 文件指针进，摘要出；禁止整包父对话历史。**

| 传什么 | 不传什么 | 为什么 |
|--------|----------|--------|
| 短任务描述、可选短备注 | 父窗口全文消息 | 省窗、缩权限与噪音 |
| 近期/点名文件路径列表 | 文件全文（除非子任务再读） | 路径几十字 vs 正文几千 token；盘是共享的 |
| — | 无限深度套娃 | 深度与步数有上限，防烧钱 |

回流：**摘要 + 产物引用**，不把子循环的工具瀑布倒进主窗。同一套循环引擎、独立消息窗、更窄工具集。

父取消应级联子任务（见 Q19）。

## Q5. 是否用 ReAct？优劣？

**结论：✅ 主循环 = `while` + Function Calling，即工程上的 ReAct。**

| | 固定工作流 | Agent 循环 |
|--|------------|------------|
| 路径 | 人事先画好 | 模型逐步决定 |
| 简单任务 | 常穿完整链 | 可一轮结束、零工具 |
| 加能力 | 改图加路由 | 注册工具 |
| 状态 | 易膨胀大对象 | 有序消息为主 |
| 可预测性 | 高 | 较低 → 用预算与评测补 |

**优**：成本延迟自适应；可中途取消；步骤可见。  
**劣**：步数膨胀、重复搜、假参数。  
**抑**：最大步数/预算、schema 门、只读缓存、超时与卡住巡检、晚步缩工具、Golden。

编排框架若用，只做检查点/中断壳，不承载业务节点图。

## Q6. 模型捏造工具参数怎么办？

**结论：✅ 执行前确定性拦住；错误结构化回灌，不让副作用发生。**

```text
tool_call →（需批则等人）→ JSON Schema
  通过 → 超时内执行
  失败 → 告知缺什么/哪错了 → 模型可改参再调
```

| 做法 | 取舍 |
|------|------|
| 再问模型「参数对吗」 | ❌ 慢，仍可能错 |
| Schema 校验 | ✅ 毫秒级、可重复 |
| 人批高风险写/执行 | ✅ 副作用晚绑定 |

口诀：**形状靠 schema；副作用靠审批与扫描；语义靠模型与证据。** Schema 拦不住「类型对但内容荒谬」。

## Q7. 多 Agent 常见模式？本系统选啥？

**结论：🔸 认真做两种：单 Agent + 工具，以及监督者–委派。**

| 模式 | 态度 | 理由 |
|------|------|------|
| 单 Agent + 工具 | ✅ 主形态 | 多数写作/短改够用；一条消息链最好懂 |
| 监督者–委派 | ✅ | 长探索隔离窗；工具可收窄；摘要回流 |
| 固定 Pipeline | ❌ | 与单循环冲突；简单任务空跑 |
| 对等总线 / 群聊式 | ❌ | 协调与权限难，调试爆炸 |
| 每轮强制评审 Agent | ❌ | 伤速率 → 用户触发核查 + 离线抽样 |

## Q8. 一万长文档做 RAG？

**结论：🔸 小库～千档「查得快」已齐；企业万档（权限、分片、运维）未完。**

### 两平面（核心）

```text
交互面：按需搜 → 只读已有索引 → 短摘录 + 引用 id
索引面：资料变更 → 异步切块 / 嵌入 → 写索引
```

查询路径 **禁止** 同步全量重建。索引空/滞后：keyword 兜底 + 滞后提示——交互继续，正确性短暂降级。

### 已有 vs 缺口

已有：混合召回、存储可换、轻量重排可开、交叉编码器默认关。  
缺口：无多租户 ACL；词法非分布式倒排；切块偏规则；缺万档持续回归。

扩库顺序直觉：权限进检索 → 索引分层 → 两级召回 SLA → 旁路升级切块 → 可选精排严超时 → 评测闸。  
铁律：**建索引 / 重精排不得回到用户第一次搜索的热路径。**

## Q9. Context / Prompt / Harness 分别是什么？

**结论：✅ 三层分工清楚；压缩按「信息损失从小到大」升级，不是「满了再摘要一次」。**

| 层 | 管什么 |
|----|--------|
| **Context engineering** | 模型看见什么、窗是否爆 |
| **Prompt** | 话术与角色 |
| **Harness** | 运行时厚度（Q14） |

### 为什么每一步组装？

一轮内多步工具会持续追加观测。只在开头压缩一次，中后段仍会超出模型上下文上限，导致请求失败或被上游截断。因此 **每调模型前** 按当前填充率重组消息窗。

### 五层压缩：机制与取舍

设计目标有两个约束同时成立：

1. **硬约束**：准备发送的输入 token 不得超过「模型窗 − 输出预留 − 系统/工具表占用」。  
2. **软约束**：尽量保留「当前任务仍需要的语义」——用户原意图、最近工作集、可再读的指针；优先丢掉 **可从磁盘/工具重取的观测副本**。

因此分层不是文案语气上的「越来越狠」，而是 **可逆性与语义损失递增**：

| # | 触发 | 机制 | 丢掉什么 / 留下什么 | 原理（为何这一层存在） |
|---|------|------|---------------------|------------------------|
| 1 | 几乎每步 | **单条工具结果预算** | 丢掉单次观测正文的超长尾部；留下截断正文 +「可再读」指针 | 单次检索/读文件即可占满窗。先做 **局部长度上界**，且内容通常仍可通过对同一路径再调用工具取回 → 损失主要是「窗内副本」，不是唯一真源 |
| 2 | 常做（与填充率无关） | **连续工具结果微折叠** | 把一段连续的多条 tool 结果收成一条指针说明；**紧跟 assistant tool_use 的配对 tool 消息保留** | 降低工具观测的 **重复占用**。必须保留配对，否则部分供应商协议要求每个 tool_call_id 都有对应 tool 消息，折叠不当会直接调用失败。本层不删除用户原话，也不改写意图 |
| 3 | 填充率 ≳ 80% | **中间历史折叠** | 保留：最早用户意图（头）+ 按 token 预算切出的最近热尾；中间段换成折叠指针（可附带工具名计数等粗线索） | 长循环里中间段多为已消费的工具史。相对用户原话与近期工作集，中间观测 **重取成本低、语义边际效用低**。折叠仍保留「曾经发生过」的占位，便于模型决定是否再搜/再读，而不是假装历史不存在 |
| 4 | 折叠后仍 ≳ 90% | **按组剔除最旧完整消息** | 从最旧一端删除整组消息（保持 assistant/tool 配对不被拆断） | 折叠只是替换中间段；若热尾本身已很大，填充率仍可能超限。此时必须做 **不可逆删除** 才能继续降占用。先删最旧，是假设时间局部性：越早的轮次对当前步决策的条件信息越弱 |
| 5 | 仍 ≳ 95% | **整窗收成结构化摘要** | 用一条摘要消息替换大段历史（可走较小模型；失败则确定性规则摘要） | 前四层仍可能压不下去（热尾过长、系统块过厚等）。摘要是 **有损压缩**：用更短的表示逼近原轨迹，使请求满足硬上下文上限。它有额外延迟/费用，且可能丢细节，故 **仅在高水位触发**，禁止每步默认摘要 |

**为何必须按这个顺序，而不是一步到位摘要？**

| 若反过来 | 后果 |
|----------|------|
| 每步都摘要 | 稳定多付一轮模型（或规则）成本；摘要误差累积；行为更抖——违反「热路径不加不必要同步模型」 |
| 先硬删再折叠 | 过早不可逆丢失；本可用指针保留的「可再读线索」没了 |
| 只截断单条、不做后面几层 | 多步之后消息条数仍线性涨，窗照样爆 |

**工作集直觉（解释保留策略，不是又一层算法）**：

| 区 | 倾向保留形态 | 例子 |
|----|--------------|------|
| 热 | 原文 | 当前约束、最近几轮、正在改的文件指针 |
| 温 | 短结构 | 阶段性结论、跨轮未完成项 |
| 冷 | 指针即可 | 很早的工具日志、已折叠中间段 |

Harness：在裸模型调用外包一层可观测、可取消、可降级的运行时机制；厚度指机制完备度，不是代码行数。约束仍是：**加厚不得拖死首 token / 取消时延**（见 Q14）。

## Q12. 保证效果下如何加速？

**结论：🔸 服从「不挡受理 / 可取消 / 可降级」。**

| 杠杆 | 原理 |
|------|------|
| 先回执再跑循环；token 流式推 | 体感卡点首先是受理与首字 |
| 只读并行；写/执行串行 | 只读无冲突 |

| 晚步缩工具表 | 减空转与提示体积 |
| 检索热路径不建索引 | Q8 |
| 网关重试纪律 | 见下 |

### 重试：什么时候试

| 规则 | 为什么 |
|------|--------|
| **尚未吐出任何 token** 才对瞬时错误重试 | 已流式再试会重复/错乱半截 |
| 已吐出再失败 → 致命，不「再碰运气」 | 失败终态不得伪装完成 |
| 退避睡眠可被取消打断 | 「再试」不能绑架「要停」 |
| 首字节/连接超时偏短 | 快失败 |
| 顺序 | 先稳调用 → 再缓存 → 再加厚上下文 |

## Q13. 为什么混合召回？怎么切块、怎么嵌？

**结论：✅ 工具中介 RAG；质量主要靠入库前结构，不靠查询时再堆模型。**

### 为什么混合？

| 通道 | 更擅长 |
|------|--------|
| 词法（如 BM25） | 专名、精确短语 |
| 向量 | 同义改写、换说法 |
| 文件 keyword | 索引空/挂时的兜底 |

**RRF**：按名次做倒数融合（经典 1-based：`1/(k+rank)`），可给通道加权——不必强行对齐两套不可比分数。文档级与段落级可并行，超时只留段落。文档命中可少量加成（先定哪一篇）；大库同质语料可切「向量偏重」剖面——**无通解，靠难查询集标定**。

### 切块

标题节优先；过长再滑窗并重叠；宽表可外挂指针，防污染同一向量。素材卡不进索引。回模型：短摘录 + 引用 id；通读再读文件。

### 嵌入拼装（易被追问）

向量若只吃正文，路径类型信息进不了语义空间，专名近邻易糊。**索引时**：

```text
嵌入输入 ≈ 路径线索 + 稀疏标签 + 正文
展示摘录 ≈ 仍主要是正文
```

标签从路径与元数据确定性抽，热路径不用 LLM 打标。切块/嵌入输入一变，索引版本必须升高并重建。

### 热路径默认不做

同步查询改写、默认同步交叉编码器、查询时建库、每轮预注入检索包。

## Q14. Harness 是什么？

**结论：✅ 决定「好不好用」的运行时厚度——不是换模型，也不是加 workflow 节点。**

```text
入口   确定性编译；命令短路；@路径预算预读；禁止 LLM 意图分类图
上下文 组装与五层压缩；组装耗时可观测
工具   白名单、审批、超时；只读并行；schema 门
模型   统一网关：参数、错误分类、未吐出才重试、首字节超时、出站脱敏
护栏   取消贯串；模型/工具/步 三层超时；卡住巡检；出站白名单
证明   事件字段、Golden、延迟门禁、离线打分/抽样
```

| 机制 | 解决 |
|------|------|
| 三层超时 | 外部依赖 hang、单步死循环 |
| 卡住巡检 | 「进程活着但很久无新事件」的假活；先告警，默认不立刻自动失败 |
| 取消 | 人要停（与超时互补） |

**厚与快**：受理优先；快失败；可打断压倒「再试」；加厚靠缓存或硬上限抵消；可测才算数；**策略不进引擎场景分叉**。

## Q16. 动态 vs 静态 Prompt？

**结论：✅ 能静不动；动态块可截断、有预算。**

| | 例子 | 为什么分开 |
|--|------|------------|
| 静态 | 场景系统提示、工具名/描述/schema | 前缀稳 → 利缓存；角色长期一致 |
| 动态 | 运行时块、弱提示、钉选、压缩后会话尾 | 随任务变；可按窗口丢 |

焦点文件全文不要焊进不可变前缀——焦点一变整段缓存失效。稳定/易变分家是加分项，设计已定、布局按排期。

## Q18. 为什么区分静态 / 动态长期记忆？

## Q19. 端到端怎么转？取消和审批有何不同？

**结论：✅ 先回执再循环；取消与审批是两种完全不同的终态语义。**

```text
发消息 → 控制面建轮/运行 → 执行面先受理
  → 少数命令可零模型短路
  → while: 组装 → 流式 FC → 审批/schema/执行
  → 结束：投影 / 摘要 / 未完成项
  → 控制面推事件 → 界面
```

旁路：索引异步、打字预热、离线抽样与打分。

| | 取消 | 审批通过 |
|--|------|----------|
| 含义 | 不要这次了 | 允许高风险副作用继续 |
| 同一运行 | **终态，不可恢复**；下一条是新轮次 | **同一运行继续** |
| 必须打断 | 退避睡眠、组装、预读、流式、工具、子委派 | — |
| 软 / 硬 | 默认尽快在检查点停；强制可立即断连 | — |
| 界面 | 可先停渲染，再对齐终态 | 「等人批」≠「停止」 |

不做「取消后再恢复同一运行」——避免状态机膨胀与半截副作用难解释。

## Q20. Function Calling 怎么运作？

**结论：✅ 意图接口在模型；校验与副作用在运行时。**

1. 场景 → 白名单 →（可再缩）工具表发给模型  
2. 模型返回 tool_calls  
3. Schema 校验  
4. 写/执行类审批  
5. 执行（检索只查不建；写出前密钥扫描）  
6. 结果回灌；引用进证据集或标未核验  

**口诀：模型出意图，运行时出事实。** 有了 FC 仍要 Harness：FC 不管参数语义对不对、该不该发生副作用、窗爆了怎么办。

## Q21. 为什么不继续固定流水线，而要重写？

**结论：✅ 旧形态用强制阶段链假装可控；新形态用单循环 + 工具 + Harness 换可控来源。**

| 旧病 | 表现 | 根因 |
|------|------|------|
| 空跑税 | 短编辑也穿分类→规划→检索→验证 | 路径人事先钉死 |
| 难扩展 | 加能力改图加路由 | 业务与编排耦合 |
| 难取消 | 停不到流式/长事务/子过程 | 检查点不在循环语义里 |
| 假活 | 无界等待 | 缺超时与卡住巡检 |
| 状态膨胀 | 大对象难测难回放 | 轨迹未收敛到消息序列 |

新可控性来自：确定性入口（场景由用户/API 指定，禁止 LLM 意图分类图）、工具与审批、预算与贯串取消、Golden 与 SLO、Harness 六面。

口诀：**加能力 = 加工具 / 加厚，不是加节点。**

## Q22. 事件、流式、投影如何分工？

**结论：✅ 事件是事实源；投影是 UI 折叠视图；推送通道归控制面独占。**

```text
执行面追加事件（工具起止、token、计划、终态…）
  → 持久化
  → 控制面拉取并推给前端
  → 投影异步刷新「当前轮视图」（不挡热路径）
```

| 原则 | 为什么 |
|------|--------|
| 执行面不对浏览器直推 | 鉴权、连接、扇出集中；故障域清晰 |
| 前端不发明阶段 | 避免本地启发式与真相打架 |
| 投影可慢一点 | UI 最终一致；索引/评价同理旁路 |
| 分层直觉 | 资源 → 命令 → 事件（事实）→ 投影（视图） |

## Q23. 会话和作品为什么必须拆开？

**结论：✅ 会话是对话单位；作品是书稿真源。换会话 / 压缩对话不拆书。**

错把会话当书：每开一会话多一棵草稿树 → 第 1–2 章在 A、第 3–4 章在 B → 同一作品被切开，续写靠碰巧记得路径。

| 层 | 语义 |
|----|------|
| 大纲 / 正式稿 / 在编草稿 | 挂在作品（工作区）下，跨会话稳定 |
| 回合快照 / 本轮触碰清单 | 审计、导出本轮；可清理 |
| 会话转录 | 只承载对话与压缩，**不拥有章节所有权** |

用户仍说「写第三章」「接着写」；系统用短作品索引（毫秒级、无同步模型）定位。压缩或新会话只影响对话窗。

## Q24. 长写作如何控 Token 又不伤效果？

**结论：🔸 磁盘是全书，窗口是当前作业面；否决同步「省钱裁判」模型。**

成本大头常是：滚长工具轨迹、误读全书进窗、无焦点通读——不是「输出正文本身」（输出往往必要）。

| 默认保留 | 理由 |
|----------|------|
| 当前章在编块 | 作业面 |
| 上一章尾（硬顶） | 衔接，不是全书 |
| 短作品索引 / 大纲目录 | 定位 |
| 相关素材钉选（有预算） | 人设文风 |
| 用户本轮点名的文件 | 意图优先于自动裁 |

可砍：更早章全文（除非点名）、过期工具瀑布、重复读同一大文件、无焦点通读全书。

**硬否决**：为首 token 同步再调模型「先摘要再写」；把「必须新开会话」当唯一省钱手段；为缓存命中率拆产品墙（如强迫章节下拉才能写）。稳定/易变分家是加分项，不能绑架交互。

## Q25. 「跑完了」为何不等于「交付对了」？

**结论：✅ 完成只说明循环收束；产物正确靠范围显式、可审差异、观测不说谎。**

| 事故型思维 | 纠正 |
|------------|------|
| completed = 稿件正确 | 完成是执行终态；引用可能未核验、导出范围可能错 |
| 用文案关键词猜「该不该检索」 | 以真实工具调用为准，勿扭曲观测 |
| 导出「当前全部」过宽 | 基于本轮触碰清单或用户显式章节集 |
| 模型直接落盘正式稿 | 主路径倾向：**提出改动 → 人审 → 控制面命令落盘** |

与写文件审批同一哲学：**副作用晚绑定、可拒绝**（Diff-first）。

## 附录 C — 深挖追问速查

| 追问 | 骨架 |
|------|------|
| 和固定业务图比？ | 旧强制阶段空跑；新单循环+工具；框架只留检查点/中断 |
| 检索为何不预注入？ | 占窗、拖首 token、破坏缓存；润色应能零搜索 |

| 微折叠 vs 中间折叠 vs 剔除 vs 摘要？ | 降重复占用且保协议配对 → 替换中间工具史为指针 → 不可逆删最旧组 → 有损摘要以满足硬上限 |

| 受理回执 vs 首 token？ | 「系统接住了」vs「模型开始出字」；都要有目标 |
