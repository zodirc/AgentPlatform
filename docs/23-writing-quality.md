# 23 — 写作质量与文风优化（场景导向执行方案）

> **性质**：可排期的设计 / 执行方案；**本文本身不改代码**。  
> **问题**：在已有「资料库上传 → 素材卡 pin → RAG / 成稿工具」之上，如何让写作更规范、少 AI 味、排版与成稿质量更稳，同时不牺牲交互速率，并正确处理 RAG 切入与 prompt cache？  
> **约束继承**：[17-execution-plan.md](17-execution-plan.md) R1–R5；[11](11-product-experience.md) TTFB / 首 token；[ADR-005/006/014](adr/)；[14-model-harness.md](14-model-harness.md)「**先 cache 再加厚**」。  
> **现状基线**（2026-07）：writing `system.md` 已规定卡片优先、检索预算、引用与交付；`sources/cards/` 角色/情节/文风卡可 **pin**（`cards.pinned`）；`search_sources` 按需；AH2 prompt cache 打在 **稳定 system 前缀 + tools**；离线 `offline/rubric.py` 仅为启发式，**非**热路径裁判。**WQ0–WQ4 已落地**（见 [`24`](24-writing-quality-execution.md)）：库存确定性选卡与预算、style 模板与 Samples/Don't、export 结构 lint、`/polish`/`/outline` 确定性展开、golden + 离线 rubric。  
> **关联**：多模态见 [`21`](21-multimodal-design.md)；Skills 去留见 [`22`](22-skills-layer.md)（**质量主路径不依赖 Skills 层**）；**票级落地见 [`24-writing-quality-execution.md`](24-writing-quality-execution.md)**。  
> **本文结论（摘要）**：质量优化按 **用户任务场景** 拆杠杆，而不是「再加一层默认裁判 / 默认 Skills / 每轮强制 RAG」。**多重微调（意图→大纲→成稿→改稿→文风→排版）组织成用户驱动的多 Turn「pass 走廊」**（§1.1），跨 pass 复用同一稳定前缀以吃满 cache（§5）；优先加厚 **style 素材卡 + 稳定前缀**；RAG 只在 **成稿取证** 与 **核对补证** 两个点切入（§4）；去 AI 味靠 **文风约束与少样本（卡内）**，禁止 Turn 末强制 LLM judge。详见 §0 / §9。

## 0. 去留与优先级（先读）

### 0.1 一句话

「写得好」不是单一指标：证据真、人物稳、文风像人、结构干净、排版可交稿——对应 **不同杠杆**。  
当前栈已经具备正确骨架（卡 pin > 用户句 > RAG）；缺口主要在 **文风卡产品化不够厚、排版/交稿规范未产品化、质量反馈未进可回归面**，而不是缺一条新的固定写作 pipeline。

### 0.2 决策表

| 方向 | 近期末？ | 理由 |
|------|----------|------|
| 每轮强制 retrieval | **否** | 改稿/润色被检索绑架；违「Skip retrieval」已有纪律与速率 |
| Turn 末强制 critique / judge Agent | **否** | [`17`](17-execution-plan.md) 已否决；多一轮 + 延迟 |
| 默认预注入 Skills 扛质量 | **否** | [`22`](22-skills-layer.md)；主路径应在卡 / system / 工具 |
| 加厚 **style 卡**（禁词、句式、标杆段落） | **是（优先）** | 已有 pin 通道；进稳定前缀 → **与 cache 同向** |
| 交稿/排版 **确定性模板**（export 前后） | **是（次优）** | 少模型参与；可测；不挡 TTFB |
| RAG 只服务证据，检索结果 **不进 cache 前缀** | **保持** | 可变后缀；保护前缀命中 |
| 用户触发的「去 AI 味润色」单次 pass | **可选（后）** | 显式命令/按钮；不计默认热路径 |
| **pass 走廊**（多重微调 = 用户驱动多 Turn，复用同一前缀） | **是（组织原则）** | §1.1；零新机制，纯产品用法 + 确定性约定 |
| 前缀确定性规则 C1–C5 落为断言 | **是（cache 地基）** | §5.2；微调越多次 cache 收益越大 |
| 写作质量 golden / 抽检 rubric 加厚 | **是（并行可证明）** | Proof 面；可离线 |

### 0.3 与素材库 / 素材卡的关系

```text
上传资料 → sources/（可索引）     → RAG：事实、细节、引用
整理卡片 → sources/cards/         → Pin：人物 / 情节骨架 / 文风（本方案加厚点）
成稿工具 → draft_section / patch  → 交付与 diff
```

**质量 ≠ 资料越多。** 资料解决「写得对不对得上材料」；文风卡与规范解决「写得像不像、排得齐不齐」。

## 1. 从哪些「场景」思考优化（用户任务切片）

不要从技术名词倒推。按用户实际在干什么切片：

| 场景 ID | 用户在做什么 | 质量痛点 | 主杠杆 | RAG？ | 速率注意 |
|---------|--------------|----------|--------|-------|----------|
| **W1 立人设/定调** | 建角色卡、文风卡 | 后续章节口吻漂移 | style/character **卡** | 否（或极少） | pin 进稳定前缀，利于 cache |
| **W2 据材料新写** | 「按资料写第 N 章」 | 幻觉、假引用、空泛 | RAG + cite 纪律 + 卡约束 | **要**（预算 2–3 次） | 检索异步索引；热路径只查 |
| **W3 局部改稿** | 改一段、收 diff | AI 套话、语气突变 | style 卡 + `propose_patch` | **通常跳过** | 遵守 system「Skip retrieval」 |
| **W4 去 AI 味润色** | 「更口语 / 少排比」 | 仍像模型 | 卡内 negative + 少样本；或用户触发 polish | **跳过** | 禁止默认同轮「写完再审」 |
| **W5 结构/大纲** | 调章节 | 结构乱 | `update_outline` + plot 卡 | 否 | 轻 |
| **W6 交稿排版** | 导出、标题层级、空行 | 排版不统一 | **确定性** export 模板 / lint | 否 | 无模型或极少 |
| **W7 核对引用** | 引用是否指得到 | 假 cite | `check_citation` / `/verify` | 按需 | verify 异步或用户触发 |

**产品含义**：同一套 writing Profile，用 **任务类型** 决定动哪根杠杆；Intake 保持确定性（slash / 显式按钮），**不**用 LLM 分类「这是 W4」。

可选产品入口（不增意图节点）：

- `/polish` 或 UI「润色（去套话）」→ 仅 W4  
- `/verify`（已有）→ W7  
- 资料库 / 卡片编辑器（loop 外）→ W1  

### 1.1 多重微调：「pass 走廊」（用户驱动的多 Turn 精修）

一个真实写作任务本质上要经过多重微调：**整体意图 → 大纲 → 成稿 → 改稿 → 文风 → 排版**。  
本方案把它组织为 **pass 走廊**：每个 pass = **一个用户显式发起的 Turn**（或 slash），走同一条 loop、同一套稳定前缀；**平台不自动串联 pass**。

```text
意图 pass   「这章要写李云龙动摇」        → 对话/update_plan（W5 前置，可与大纲合一）
大纲 pass   /outline 或「先列大纲」        → update_outline（W5）
成稿 pass   「按资料写第 N 章」            → RAG 取证 + draft_section（W2）
改稿 pass   「第三段重写，弱化说教」        → propose_patch，0 检索（W3）
文风 pass   /polish                        → 卡约束润色，0 检索（W4）
排版 pass   导出                           → export 模板 + lint，无模型（W6）
核对 pass   /verify                        → 确定性核对，缺证再补检索（W7）
```

走廊三条硬规则（对齐速率与 cache）：

1. **每 pass 一 Turn，用户驱动**：禁止平台自动「成稿完接文风、文风完接排版」；用户可跳过任意 pass。  
2. **全走廊同一前缀**：各 pass 共享 `system + pinned cards + tools`（§5.1）；pass 差异只体现在 **用户句 / slash 展开**（可变后缀）→ 走廊越长，cache 收益越大，而不是越慢。  
3. **每 pass 有默认检索开关**（§4.2）：仅成稿/核对可检索，其余 pass 默认 0 次 `search_sources`——这是可断言的确定性行为，不是建议。

**为什么不是 pipeline**：pass 顺序、取舍、重复次数全由用户决定（比如只跑改稿 pass 五次）；平台只保证每个 pass 快、可 diff、可回归。这与 ADR-005 一致：多 pass 是 **产品用法**，不是执行图。

## 2. 质量维度 × 机制映射

| 维度 | 用户感知 | 错误做法 | 推荐做法 |
|------|----------|----------|----------|
| **规范**（术语、人称、世界观） | 前后矛盾 | 每轮塞长 system | character/plot **卡 pin**；禁项写进卡 |
| **少 AI 味** | 套话、排比、空洞形容词 | 末尾再开裁判模型 | style 卡：**禁词表 + 句式偏好 + 1～3 段用户真文少样本** |

| **交付诚实** | 说写完但文件不对 | （历史问题见 [`13`](13-writing-delivery-issues.md)） | 保持 export/`section_ids` 纪律；golden |

## 3. 速率（R1–R5）红线

| 红线 | 写作质量方案中的含义 |
|------|----------------------|
| **R1** | 建卡、索引、rubric、模板校验 **不得**推迟 `turn.accepted` |
| **R2** | 首 token 前禁止「为了文风再问一小模型」；禁止默认同步 judge |
| **R3** | pin 卡：读盘 + 截断，毫秒级；卡正文硬顶（见 §5） |
| **R4** | 资料索引、批量评分、全库风格分析 → 异步 / 离线 / 用户触发 |
| **R5** | 每期质量改动至少：卡 pin 单测、或 writing golden、或「改稿跳过检索」断言 |

**禁止的「质量换速度」模式**：

```text
写完 → 自动 delegate(fact_checker|style_judge) → 再改一稿再给用户
```

允许的模式：

```text
用户显式 /polish 或点「润色」→ 单次 Turn（可无 RAG）→ diff 审阅
```

## 4. RAG 怎么切入（服务证据，不服务文风）

### 4.1 原则

| | RAG（sources 索引） | 素材卡（cards pin） |
|--|---------------------|---------------------|
| 擅长 | 细节、专名、可引用片段 | 口吻、禁区、关系、文风少样本 |
| 失败模式 | 噪声进稿、简单改稿也检索 | 卡过大挤爆窗口、乱 pin |
| 与质量 | **保真** | **文风 / 规范** |

### 4.2 切入策略：全走廊只有两个 RAG 切入点（非必须）

RAG 在写作里 **不是默认在场**，只允许两个切入点，其余 pass 默认关：

| 切入点 | pass | 触发 | 预算 |
|--------|------|------|------|
| **取证** | 成稿（W2） | 草稿需要材料细节 / 用户要引用 | 每题 ≤2–3 次；低分改 `read_file` |
| **补证** | 核对（W7） | `/verify` / `check_citation` 发现 cite 指不到 | 按缺口补，不全库重扫 |

```text
意图/大纲/改稿/文风/排版 pass → 默认 0 次 search_sources（可断言）
成稿 pass                    → 允许，预算内
核对 pass                    → 确定性检查为主，缺证再检索
```

「非必须」的含义：即使成稿 pass，自由创作（不引用 `sources/`）也可全程 0 检索——RAG 是证据工具，不是质量必经步骤。

### 4.3 与「提高质量」的常见误区

- **误区**：多检索 → 少 AI 味。  
  **事实**：检索片段是材料口吻，不自动变成作者口吻；过度检索还增加套话拼接感。  
- **误区**：把检索命中预注入每轮 system。  
  **事实**：破坏 cache、拖慢首 token；应用工具结果进 messages（可变后缀）。

## 5. Prompt cache 与加厚顺序（对齐 AH2）

[`14`](14-model-harness.md)：**先 cache，再加厚。**

### 5.1 稳定前缀（宜 cache）

```text
writing system.md
+ 本 Turn pin 的 cards（截断后）
+ tools 定义
```

文风/人物卡加厚应落在这里，并：

- **按 kind 预算**：如 style ≤ A 字、character 合计 ≤ B、全 pin ≤ C（具体数字落地时写入 settings）  
- **变更低频**：卡在 loop **外**编辑；同一 Session 多 Turn 尽量 **同一前缀** → 命中率高  
- UI 已有「本轮写定」可观测：继续暴露 pin 集合，避免 silently 变前缀

### 5.2 前缀确定性规则（cache 命中的硬约定）

cache 命中率不取决于「打没打 `cache_control`」，而取决于 **前缀字节级稳定**。落地时按以下确定性规则执行（每条可测）：

| # | 规则 | 断言方式（落地时） |
|---|------|--------------------|
| C1 | **pin 集合选择确定性**：同一 cards 目录状态 + 同一预算 → 选出的 pin 集合与顺序 **恒同**（排序键固定：kind → path；**不依赖用户句**，以保证走廊 C3） | 单测：同库存两次选卡（甚至不同 message）序列相等 |
| C2 | **截断确定性**：卡超预算的截断只依赖（卡内容, 预算），不依赖时间/随机 | 单测：截断幂等 |
| C3 | **走廊内前缀不漂移**：同一 Session 连续 pass 之间，若 cards 未编辑，则 system+cards+tools 前缀 **字节一致** | golden / 日志：连续两 Turn 前缀 hash 相等 |
| C4 | **卡编辑 = 显式换前缀**：仅当用户改卡时才允许前缀变化，UI 通过「本轮写定」可见；平台不得在 pass 中途自动增删 pin | 事件 `cards.pinned` payload 对比 |
| C5 | **易变项一律后缀**：runtime_context（step 计数）、plan_hint、检索结果、草稿轨迹不得进前缀 | 代码审查项 + 结构断言 |

> **实现注记（WQ0）**：选卡已改为 **库存确定性**（`message` 保留 API 兼容但不参与打分），见 [`24`](24-writing-quality-execution.md) §1.2；`cards.pinned` 携带 `prefix_hash` / `budget` / `truncated` / `dropped`。

> C3 是「pass 走廊」（§1.1）的 cache 红利来源：意图→大纲→成稿→改稿→文风多个 Turn 反复命中同一前缀，**微调越多次、平均每 Turn 输入成本越低**，而不是线性变贵。

### 5.3 可变后缀（不宜进 cache 前缀）

```text
用户本句 / slash 展开
+ search_sources / read_file 的 tool_result
+ 本 Turn 草稿轨迹
+ runtime_context / plan_hint（step 相关，天然易变）
```

### 5.4 加厚检查清单（合并前）

1. 新增常驻字数是否可被 cache 抵消？  
2. 是否把易变 RAG 误放进前缀？  
3. `context.reported` / usage cache 字段是否仍可观测？  
4. 简单 W3 任务 golden：检索次数仍可为 0？  
5. C1–C5 是否仍成立（尤其新增任何「动态选卡 / 动态附言」逻辑时）？

## 6. 推荐方案架构（开闸后怎么做）

### 6.1 分层（不改 loop）

```text
┌─────────────────────────────────────────────────────────┐
│ Loop 外（产品）                                          │
│  资料上传 / 卡片编辑器 / 文风向导 / 交稿模板库              │
└─────────────────────────────────────────────────────────┘
                            │ pin / 配置
                            ▼
┌─────────────────────────────────────────────────────────┐
│ Turn Intake（确定性）                                    │
│  选卡 pin → 截断 → 拼稳定 system 前缀                     │
│  slash：/polish /verify（已有）…                         │
└─────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────┐
│ AgentEngine（不变）                                      │
│  按需 RAG 工具 · draft/patch · 可选用户触发润色 Turn      │
└─────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────┐
│ 交付（确定性优先）                                        │
│  export 模板 · markdown 结构校验 · diff 人工终审          │
└─────────────────────────────────────────────────────────┘
```

### 6.2 Style 卡内容约定（质量主杠杆）

建议卡模板（loop 外生成，可向导）：

| 块 | 作用 |
|----|------|
| **Voice** | 人称、语气、时代感 |
| **Do** | 鼓励的句式 / 节奏（短句、对话比例…） |
| **Don't（去 AI 味）** | 禁词与禁结构（「在这个时代」「不禁」「充满了」清单——按作品定制） |
| **Samples** | 1～3 段 **作者或已定稿** 原文（少样本；计入卡预算） |
| **Format** | 标题层级、对话标点、场景分隔（与 W6 模板对齐） |

Pin 优先级保持：`pinned cards > 用户句 > search_sources`。

### 6.3 排版 / 规范（确定性）

| 手段 | 说明 |
|------|------|
| Export profile | 如 `novel-zh` / `essay`：标题、空行、禁止 HTML |
| 结构 lint | 导出前检查 AT 层级、空 section；失败明确报错（对齐交付诚实） |
| 大纲源 | `outline.md` 为结构真源，避免模型自由发挥层级 |

### 6.4 可选：用户触发润色 Turn（文风 pass 的产品入口）

- 入口：`/polish` 或按钮；**新 Turn**，实现为 **Intake 确定性 slash 展开**（把润色约束追加进 **用户消息**，如 `[polish] 只改文风与节奏；禁改专名、情节、[cite:*]；逐段 propose_patch`）——**不改 system 前缀**，保护 C3 cache 命中  
- 默认 **不调用** `search_sources`（对齐 §4.2：文风 pass 0 检索）  
- 产出走 `propose_patch`，用户 diff 接受  
- **不**自动串联在每次 `draft_section` 之后  

同理可加 `/outline`（大纲 pass）：展开为「只产出/修改 outline.md，不写正文」的用户侧附言。两者都是 **确定性文本展开**，与 `/compact` `/verify` 同族，不引入 LLM 路由。

### 6.5 与 Skills（22）边界

| 质量能力 | 放哪 |
|----------|------|
| 常驻文风 / 禁词 / 少样本 | **style 卡**（本方案） |
| 主路径检索与交付纪律 | **system.md**（已有） |
| 极低频文体包（学术 GB/T 等） | 将来才考虑 Skills / workspace 包（22 开闸后） |

## 7. 分期建议

| 期 | 主题 | 对应动作（§9.1） | 速率门槛 | 场景覆盖 |
|----|------|------------------|----------|----------|
| **Q0** | cache 地基 + 文风卡模板：pin 预算硬顶、前缀稳定断言、卡模板/向导 | D1 D2 D3 | 前缀可控；cache 不回归 | W1/W4 基础 + 全走廊 |
| **Q1** | Don't 清单 + Samples 导入（从已定稿章节抽） | D1 深化 | 卡总预算；超限截断可测 | W3/W4 |
| **Q2** | Export 模板 + markdown 结构 lint | D6 | 无模型；失败快 | W6 |
| **Q3** | `/polish` `/outline` 确定性展开 + golden（跳过 RAG、前缀不变） | D4 D5 | 显式触发；简单任务无自动 polish | W4/W5 |
| **Q4** | 写作质量 golden 集（文风禁词、改稿 0 检索、检索预算、引用诚实）+ 离线 rubric 加厚（**不进热路径**） | D4 D7 收口 | Proof only | 全场景可证明 |

依赖：

```text
Q0 → Q1（内容）
Q0 → Q2（交付，可并行）
Q1 → Q3（有文风约束再润色才有意义）
Q0～Q2 → Q4（有行为可断言再黄金化）
```

**相对优先级**：Q0–Q2 优于 Skills（22）；与多模态（21）可并行（正交）。若资源只能选一个写作向：**先 Q0/Q1（文风卡）**。

## 8. 非目标（否决）

| 项 | 原因 |
|----|------|
| 固定「检索→写作→审校」pipeline | ADR-005；简单任务变慢 |
| 每轮自动 style judge / 多 Agent 互评 | R2；[`17`](17-execution-plan.md) |
| 把 RAG 命中写入 cache 前缀 | 击穿 AH2 |
| 用 Skills 默认 1.2K 目录代替文风卡 | [`22`](22-skills-layer.md) |
| 热路径重模型打分润色 | R3/R4；rubric 仅离线/抽检 |
| 「一个万能质量 system 附录」无限加长 | 挤 cache、抢注意力；应拆入卡与模板 |

## 9. 开闸条件与当前推荐动作

### 9.1 确定性动作清单（值得且可做；每条附验收断言）

以下动作全部满足：不进热路径新模型调用、行为确定性、落地时可用单测或 golden 断言。

| # | 动作 | pass / 场景 | 验收断言（落地时） |
|---|------|-------------|--------------------|
| D1 | 产品化 **style 卡模板**（Voice / Do / Don't / Samples / Format） | 文风（W1/W4） | ✅ 模板 + Samples 导入 + `dont_enabled` |
| D2 | 设定并文档化 **pin 预算** 与超限策略（按 kind 截断 vs 丢弃，规则固定） | 全走廊 | ✅ C1/C2；`cards.pinned` 截断/丢弃标记 |
| D3 | **前缀稳定性**落为断言：同 Session 连续 Turn（卡未变）前缀 hash 相等 | 全走廊 cache | ✅ C3 单测 / `prefix_hash` |
| D4 | 改稿/文风 pass golden：**0 次** `search_sources` | W3/W4 | ✅ `writing.12` / `writing.13` + `forbidden_names` |
| D5 | `/polish` `/outline` 作为 **确定性 slash 展开**（用户侧附言，不动 system） | 文风/大纲 | ✅ `expand_writing_slash` |
| D6 | Export 结构校验（标题层级、空 section、`section_ids` 完整性），失败明确报错 | 排版（W6） | ✅ `export_lint` + `profile` |
| D7 | 成稿 pass 检索预算写死可测（每题 ≤N 次，超出 system 已有条文 + 观测计数） | 成稿（W2） | ✅ runtime 预算 + `writing.05` `max_calls` |

推荐顺序：**D2/D3（cache 地基）→ D1（内容）→ D4/D5（pass 纪律）→ D6/D7（交付与证据）**。D2–D5 合起来即「多重微调不变慢、反而更便宜」的最小闭环。

### 9.2 需要证据再做

- 自动 `/polish` 串联、默认多步润色  
- 工作区外「全球文风模型」或每用户向量文风库（易贵、易慢）  

### 9.3 当前推荐口径

> **用素材卡（尤其 style）承载规范与去 AI 味；用 RAG 承载证据；用确定性模板承载排版；用显式润色与 diff 承载终修；用 cache 保护稳定前缀。**  
> 不在热路径上堆裁判，不把质量问题误诊成「再多检索一次」。

## 10. 风险登记

| 风险 | 缓解 |
|------|------|
| Style 卡过大 → 首 token / 费用升 | 硬顶；Samples 限段数；AH2 抵消 |
| Don't 清单过拟合 → 文风僵硬 | 卡按作品；允许用户关某张卡 |
| 润色 Turn 改掉事实 | polish 系统附言：禁止改专名与 cite；高风险走 verify |
| 质量黄金主观 | 先断言「禁词/结构/检索次数」等客观项；主观分仅离线 |
| 与多模态同时加厚前缀 | [`21`](21-multimodal-design.md) 附件走引用；图文勿进 cache 前缀 |

## 12. 总结

| 角度 | 结论 |
|------|------|
| **从哪类场景想** | W1–W7 任务切片；**多重微调 = 用户驱动的 pass 走廊（§1.1）**，不是自动 pipeline |
| **速率** | 禁默认审校链与强制检索；每 pass 一 Turn、检索开关确定；pin 与模板走热路径但须预算 |
| **RAG** | 全走廊仅 **取证（成稿）/ 补证（核对）** 两个切入点，且非必须；其余 pass 0 检索可断言 |
| **Token cache** | 前缀确定性规则 C1–C5（§5.2）；走廊内多 pass 复用同一前缀 → **微调越多，平均每 Turn 越便宜** |
| **是否值得** | **值得**：§9.1 D1–D7 全部确定性、可断言、不动 loop；主路径是文风卡 + 模板 + 可证明，不是新 pipeline / Skills / 每轮 RAG |

**最终口径**：写作质量优化 = **pass 走廊承载多重微调** + **卡与模板产品化** + **RAG 限定两切入点** + **前缀确定性吃满 cache**；与 21/22 并列进设计库存，落地按 §7 Q0（D2/D3 地基）起排，不抢「为质量而质量」的慢路径。
